## 1. MCPとは何か

### AIツール統合のオープンソース標準

MCP（Model Context Protocol）は、AIモデルが外部ツールやデータソースと安全かつ標準的に接続するための**オープンソース標準プロトコル**です。AnthropicがClaude Desktopで最初に採用し、現在は業界全体で広く使われるオープン標準となっています。

> **公式ドキュメントより**: "MCP servers give Claude Code access to your tools, databases, and APIs."

従来、AIアシスタントに外部データを渡すにはプロンプトにコピー＆ペーストするか、カスタム統合を構築するしかありませんでした。MCPは「AIが外部ツールと話す共通言語」を提供することで、この問題を解決します。

### MCPでできること

公式ドキュメントでは、MCPを活用した以下のようなワークフローが紹介されています。

| ユースケース | 具体例 |
|------------|--------|
| Issue trackerからの実装 | 「JIRAのENG-4521を実装してGitHubにPRを作成して」 |
| モニタリングデータ分析 | 「SentryとStatsigを確認して機能の使用状況を調べて」 |
| データベース照会 | 「PostgreSQLから今月の売上トップ10を取得して」 |
| デザイン統合 | 「SlackのFigmaデザインに基づいてメールテンプレートを更新して」 |
| ワークフロー自動化 | 「10人のユーザーをフィードバックセッションに招待するGmailの下書きを作成して」 |

これらはすべて、MCPサーバーを通じてClaudeが外部サービスに直接アクセスすることで実現されます。一つのプロンプトで複数サービスをまたいだ操作が完結する点が従来との大きな違いです。

### なぜMCPが重要なのか

#### 従来のワークフロー（MCPなし）

```
1. GitHubを開いてイシューの内容をコピー
2. Claudeのチャットに貼り付け
3. 回答を受け取ってGitHubに戻って実装
4. 変更をコミットしてPRを手動作成

→ 手動作業が多く、コンテキストスイッチが頻繁に発生する
```

#### MCPを使ったワークフロー

```
「JIRAのENG-4521を実装してGitHubにPRを作成して」

→ ClaudeがJIRAからイシュー情報を取得
→ コードを実装
→ GitHubにPRを自動作成
→ すべて一つのプロンプトで完結
```

#### MCPの4つのメリット

1. **標準化**: 各サービスが独自の統合方法を提供する必要がない
2. **セキュリティ**: MCPサーバーが適切なアクセス制御を実装できる
3. **再利用性**: 一度作ったMCPサーバーは任意のMCP対応AIツールで使える
4. **拡張性**: 新しいツールの追加が簡単

## 2. MCPの基本概念

MCPサーバーは3つの要素を提供します。それぞれの役割を理解することがMCPを効果的に使う鍵です。

### ツール（Tools）

ツールはClaudeが**実行できるアクション**です。関数呼び出しのように、名前・引数・戻り値が定義されています。

```
github MCPサーバーのツール:
- create_issue（イシューを作成）
- create_pull_request（PRを作成）
- search_repositories（リポジトリを検索）
- list_pull_requests（PRの一覧を取得）
```

Claudeはユーザーのリクエストに基づいて、どのツールを呼び出すべきかを自律的に判断します。「PRを作成して」と言えば `create_pull_request` を、「最近のイシューを見せて」と言えば `list_issues` を選択します。

### リソース（Resources）

リソースはClaudeが**参照できるデータ**です。`@`メンションの形式で直接指定できます。

```
@github:issue://123          # GitHubイシュー#123
@postgres:schema://users     # usersテーブルのスキーマ
@docs:file://api/auth        # APIドキュメント
```

リソースはツールとは異なり、「データを取得する」操作に特化しています。状態を変更するアクションはツールが担います。

### プロンプト（Prompts）

MCPサーバーが提供するプロンプトは、Claude Code内で`/`コマンドとして利用できます。

```
/mcp__github__list_prs       # PRの一覧を取得するプロンプト
/mcp__github__pr_review 456  # PR#456のレビューテンプレート
```

プロンプトは「よく使うリクエストのテンプレート」です。毎回同じ指示を入力する手間を省き、最適化された問い合わせを実行できます。

### 3つの要素の関係

```
MCPサーバー
├── ツール（Tools）      → Claudeが「実行する」もの（アクション）
├── リソース（Resources） → Claudeが「参照する」もの（データ）
└── プロンプト（Prompts） → Claudeが「使う」テンプレート（/コマンド）
```

## 3. MCPのアーキテクチャ

### 全体構成

MCPの通信構造を以下の図で確認しましょう。

```
┌─────────────────────────────────────────────────────┐
│                  Claude Code                          │
│  ┌──────────────────────────────────────────────┐   │
│  │          Claude (LLM)                         │   │
│  │  - ツールを選択・呼び出す                      │   │
│  │  - 結果を解釈して次の行動を決定                │   │
│  └──────────────────────────────────────────────┘   │
│           ↕ MCP Protocol                             │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐            │
│  │ GitHub   │ │PostgreSQL│ │  Sentry  │  MCPサーバー│
│  │  MCP     │ │  MCP     │ │  MCP     │            │
│  └──────────┘ └──────────┘ └──────────┘            │
└─────────────────────────────────────────────────────┘
       ↕               ↕               ↕
  GitHub API    PostgreSQL DB    Sentry API
```

Claude Code（ホスト）がMCPプロトコルを通じて複数のMCPサーバーと通信し、各MCPサーバーがそれぞれの外部サービスへアクセスします。Claudeは必要なツールを自律的に選び、結果を解釈して次のアクションを決定します。

### トランスポート方式

MCPは3つの通信方式をサポートしています。

| 方式 | 説明 | 用途 | 推奨度 |
|-----|------|------|--------|
| **HTTP** | HTTPリクエストで通信 | リモートサーバー・クラウドサービス | 推奨 |
| **stdio** | 標準入出力でプロセス間通信 | ローカルプロセスとして動作するサーバー | ローカル用途で推奨 |
| **SSE** | Server-Sent Events | レガシーリモートサーバー | 非推奨 |

**使い分けの基本方針**:
- クラウドサービス（GitHub, Notion, Sentry等）への接続 → **HTTP**
- ローカルで動作するサーバー（データベース接続、ファイル操作等） → **stdio**
- SSEは既存のレガシーサーバーのみに使用。新規設定ではHTTPを選択する

### Dynamic Tool Updates

Claude Codeはリアルタイムのツール更新をサポートしています。MCPサーバーが`list_changed`通知を送ると、Claude Codeは切断・再接続なしで利用可能なツールを自動更新します。これにより、サーバー側でツールを追加・変更しても、Claude Codeを再起動する必要がありません。

## 4. MCP vs REST API vs GraphQL

MCPと従来のAPI連携の違いを理解することは重要です。

| 比較軸 | MCP | REST API | GraphQL |
|-------|-----|---------|---------|
| **設計の目的** | AIとツールの接続 | クライアント-サーバー通信 | データ取得の効率化 |
| **接続方法** | AIが自律的に選択 | 開発者が明示的に呼び出す | 開発者が明示的に呼び出す |
| **標準化** | AIツール向けオープン標準 | HTTP上の慣例 | Facebookが開発したクエリ言語 |
| **コンテキスト共有** | セッションを通じて保持 | ステートレス | ステートレス |
| **AI向け最適化** | ツール・リソース・プロンプトの概念 | なし | なし |

REST APIやGraphQLは「開発者がコードで呼び出す」ためのものです。MCPは「AIが自律的に呼び出す」ためのものです。Claudeは MCPサーバーのツール定義を見て、ユーザーのリクエストに最適なツールを選択・実行します。

**例えば**: REST APIでGitHubイシューを取得するなら、開発者が`GET /repos/{owner}/{repo}/issues/{number}`を呼ぶコードを書きます。MCPなら、Claudeが「イシュー#123の内容を教えて」というリクエストから`get_issue`ツールを自動選択して呼び出します。

## 5. MCPの活用パターン

### パターン1: 単一サービスの操作

```
# GitHubだけを使うパターン
「PR #456をレビューして改善点を提案してください」
→ GitHub MCP → PRの内容取得 → レビューコメント
```

### パターン2: 複数サービスの連携

```
# JIRA + GitHub を連携するパターン
「JIRAのENG-4521の機能を実装してGitHubにPRを作成して」
→ JIRA MCP → イシュー取得 → コード実装 → GitHub MCP → PR作成
```

### パターン3: データ分析ワークフロー

```
# Sentry + PostgreSQL を連携するパターン
「Sentryの直近エラーを確認し、影響を受けたユーザー数をDBで調べて」
→ Sentry MCP → エラー取得 → PostgreSQL MCP → クエリ実行 → 分析結果
```

### パターン4: 自動化ワークフロー

```
# Slack + JIRA + GitHub を連携するパターン
「Slackで報告されたバグをJIRAに登録して、修正してPRを作成して」
→ Slack MCP → バグ詳細取得 → JIRA MCP → イシュー作成
→ コード修正 → GitHub MCP → PR作成
```

MCPの真の力は、複数のサーバーを組み合わせることで、従来は人間が手動で行っていた複雑なワークフローを自動化できる点にあります。

## 6. MCPエコシステムの全体像

### MCPサーバーのカテゴリ

現在、MCPサーバーは以下のカテゴリで提供されています。

| カテゴリ | 代表的なサーバー | 主な用途 |
|---------|----------------|---------|
| ソース管理 | GitHub, GitLab | PR作成、イシュー管理、コードレビュー |
| データベース | PostgreSQL, SQLite | クエリ実行、スキーマ確認、データ分析 |
| プロジェクト管理 | Linear, Jira, Asana, Notion | タスク管理、ドキュメント操作 |
| コミュニケーション | Slack, Gmail | メッセージ送受信、通知 |
| 監視・エラー追跡 | Sentry | エラー分析、スタックトレース確認 |
| ブラウザ自動化 | Playwright | テスト自動化、スクレイピング |
| インフラ | Vercel, Firebase, Supabase | デプロイ管理、データベース操作 |
| デザイン | Figma | デザインデータの読み込み、実装 |

詳細はLesson 3「主要MCPサーバー一覧」で解説します。

### MCPとClaude Codeの拡張機能の関係

Claude Codeには複数の拡張機能があり、MCPはその一つです。

```
Claude Code の拡張機能:
├── MCP         → 外部ツール・APIとの接続（本章のテーマ）
├── Skills      → カスタムコマンドの定義（/コマンド）
├── Hooks       → ライフサイクルイベントの自動処理
├── Plugins     → 上記をパッケージ化して配布
├── Chrome連携  → ブラウザ操作の自動化（Lesson 5）
└── Slack連携   → Slackからのタスク委譲（Lesson 6）
```

MCPはこれらの中で最も汎用性が高い拡張機能です。Skillsがプロンプトテンプレート、Hooksがイベント駆動の自動化であるのに対し、MCPは外部サービスとの双方向通信を実現します。

### MCPの成長と将来

MCPはAnthropicが提唱したオープン標準ですが、現在は業界全体で採用が進んでいます。GitHub、Notion、Sentry、Asanaなどの主要サービスが公式MCPサーバーを提供しており、エコシステムは急速に拡大しています。

MCPサーバーは一度開発すれば、Claude Codeだけでなく、他のMCP対応AIツールでも再利用できます。これはMCPが特定のAIツールに依存しないオープン標準であることの大きなメリットです。

## よくある質問

**Q: MCPサーバーは自分でインストール・管理する必要がありますか?**
A: はい。MCPサーバーは`claude mcp add`コマンドで追加します。リモートHTTPサーバー（GitHub, Notion等）は登録するだけで使えます。ローカルstdioサーバー（データベース接続等）はnpxやnodeでプロセスを起動します。

**Q: MCPサーバーを追加するとコストが増えますか?**
A: MCPサーバーのツール定義はコンテキストウィンドウを消費します。使わないサーバーはコンテキストを無駄に消費するため、不要なサーバーは`claude mcp remove`で削除することをお勧めします。

**Q: MCPとClaude Codeの組み込みツール（git, bashなど）の違いは?**
A: Claude Codeの組み込みツールはファイル操作・git・bash等の基本操作を提供します。MCPはそれ以外の外部サービス（GitHub API, データベース, Slack等）との連携を拡張するものです。両者は補完的に機能します。

**Q: セキュリティは大丈夫ですか?**
A: MCPサーバーへのアクセスにはユーザーの承認が必要です。特に`.mcp.json`（プロジェクトスコープ）のサーバーは使用前に承認が求められます。ただし、信頼できるサーバーのみを使用し、APIキーは環境変数で管理することが重要です。

**Q: MCPサーバーは同時にいくつ接続できますか?**
A: 技術的な上限はありませんが、多数のサーバーを接続するとツール定義がコンテキストを圧迫します。定義がコンテキストの10%を超えるとTool Searchが自動的に有効化され、必要なツールだけがロードされます。

**Q: MCPサーバーとChrome連携・Slack連携の違いは何ですか?**
A: MCPはプロトコルベースの汎用的な外部ツール接続です。Chrome連携はブラウザ拡張機能を通じたブラウザ操作、Slack連携はSlackからClaude Codeセッションを起動する機能です。MCPが「Claude Codeから外部サービスに接続する」のに対し、Slack連携は「外部（Slack）からClaude Codeを呼び出す」方向の連携です。

**Q: MCPはClaude Code以外のAIツールでも使えますか?**
A: はい。MCPはオープン標準プロトコルなので、MCPに対応した任意のAIツールで使えます。Claude Desktop、VS Code内のClaude Code、その他のMCP対応ツールで同じMCPサーバーを利用できます。

## まとめ

この章で学んだ重要ポイント：

- MCPはAIツールと外部サービスを接続するオープンソース標準プロトコルである
- MCPサーバーはツール（実行アクション）、リソース（参照データ）、プロンプト（テンプレート）の3要素を提供する
- トランスポート方式はHTTP（推奨）、stdio（ローカル用）、SSE（非推奨）の3種類がある
- MCPはREST APIと異なり、AIが自律的にツールを選択・呼び出せるよう設計されている
- 複数のMCPサーバーを組み合わせることで、複雑なマルチサービスワークフローを自動化できる
