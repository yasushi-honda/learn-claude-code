## この章の学習目標

この章を終えると、以下のことができるようになります：

1. デスクトップアプリのインストールとセッション開始の手順を理解できる
2. パーミッションモードを使い分け、適切な自律性レベルで Claude を運用できる
3. 並列セッション（Git worktree 分離）の仕組みを理解し、複数タスクを同時進行できる
4. アプリプレビュー、ビジュアルdiff レビュー、PR 監視の各機能を活用できる

## 1. デスクトップアプリの概要

Claude Code デスクトップアプリは、ターミナルの CLI と同じエンジンをグラフィカルインターフェースで操作するスタンドアロンアプリケーションです。

### 対応プラットフォーム

| プラットフォーム | アーキテクチャ | 対応状況 |
|---------------|-------------|---------|
| **macOS** | Intel (x64) | 対応 |
| **macOS** | Apple Silicon (ARM64) | 対応 |
| **Windows** | x64 | 対応 |
| **ARM64** | リモート接続のみ | 対応（ローカル実行は非対応） |

### CLI にはないデスクトップアプリの機能

- **ビジュアルdiff レビュー**とインラインコメント
- **ライブアプリプレビュー**（開発サーバー連携）
- **GitHub PR 監視**（自動修正・自動マージ）
- **並列セッション**（Git worktree による自動分離）
- **コネクタ**（GitHub、Slack、Linear、Notion など外部サービス連携）
- **ファイル添付**（画像・PDF をドラッグ&ドロップ）

## 2. セッションの開始とパーミッションモード

### インストール

デスクトップアプリは公式サイトからダウンロードできます。macOS では `.dmg` ファイル、Windows では `.exe` インストーラが提供されます。

インストール後、初回起動時に Anthropic アカウントでのサインインが求められます。

### セッション設定

セッション開始前に、プロンプトエリアで4つの項目を設定します：

| 項目 | 説明 | 選択肢 |
|-----|------|--------|
| **環境** | Claude が作業する場所 | ローカル / リモート（Anthropic クラウド）/ SSH |
| **プロジェクトフォルダ** | 作業対象のフォルダまたはリポジトリ | ファイルブラウザで選択 |
| **モデル** | 使用するモデル | 送信ボタン横のドロップダウンで選択 |
| **パーミッションモード** | Claude の自律性レベル | 下記参照 |

### パーミッションモードの詳細

| モード | 動作 | 推奨シーン |
|-------|------|----------|
| **Ask permissions** | ファイル編集・コマンド実行の前に都度確認 | 初回利用・重要な作業 |
| **Auto accept edits** | ファイル編集は自動承認、コマンド実行は確認 | ファイル変更を信頼する場合 |
| **Plan mode** | コードを分析して計画のみ（変更しない） | 複雑なタスクの事前確認 |
| **Bypass permissions** | 全ての承認をスキップ | サンドボックス環境のみ |

> **公式ドキュメントより**: 複雑なタスクではまず Plan mode で計画を確認し、承認後に Auto accept edits または Ask permissions で実行するのがベストプラクティスです。

## 3. 並列セッション（Git Worktree 分離）

デスクトップアプリの大きな特徴の一つが、複数のタスクを並行して実行できる**並列セッション**です。

### 仕組み

サイドバーの **+ New session** をクリックすると、新しいセッションが作成されます。Git リポジトリで作業している場合、各セッションには **Git worktree** による独立したファイルコピーが自動的に割り当てられます。

**Worktree 分離のメリット**:

- あるセッションでの変更が他のセッションに影響しない
- 各セッションが独立したブランチで作業できる
- コンフリクトのリスクなく並列作業が可能

### Worktree の設定

| 設定項目 | 説明 |
|---------|------|
| **デフォルト保存場所** | `<project-root>/.claude/worktrees/` |
| **カスタムディレクトリ** | 設定 → Claude Code → Worktree location で変更 |
| **ブランチプレフィックス** | Claude が作成するブランチに共通のプレフィックスを設定 |
| **セッションのアーカイブ** | サイドバーでホバー → アーカイブアイコンで削除 |

### 並列セッションの活用例

```
セッション 1: バグ修正
  └── fix/auth-validation ブランチ
  └── 認証バリデーションのバグを修正中

セッション 2: 新機能開発
  └── feat/user-profile ブランチ
  └── ユーザープロファイル画面を実装中

セッション 3: テスト追加
  └── test/api-endpoints ブランチ
  └── API エンドポイントのテストを追加中
```

各セッションが独立した worktree とブランチで作業するため、互いの変更が干渉しません。

> **公式ドキュメントより**: Git セッション分離には Git が必要です。Windows では Git のインストールが必須です。

## 4. アプリプレビュー

Claude が開発サーバーを起動して、コード変更の結果をブラウザプレビューで確認できる機能です。

### 対応範囲

- フロントエンドウェブアプリ（React、Vue、Next.js など）
- バックエンドサーバー（Express、Django など）
- 通常、Claude はファイル編集後に自動的にサーバーを起動

### autoVerify（自動検証）

`autoVerify` が有効な場合、Claude はファイル編集後に自動的に以下を実行します：

1. スクリーンショットを撮影
2. DOM を確認
3. 要素をクリック、フォームに入力して動作確認
4. 問題を検出した場合は自動修正

### launch.json の設定

開発サーバーの設定は `.claude/launch.json` に記述します：

```json
{
  "version": "0.0.1",
  "configurations": [
    {
      "name": "my-app",
      "runtimeExecutable": "npm",
      "runtimeArgs": ["run", "dev"],
      "port": 3000
    }
  ]
}
```

autoVerify を無効にする場合：

```json
{
  "version": "0.0.1",
  "autoVerify": false,
  "configurations": [...]
}
```

### launch.json の設定フィールド

| フィールド | 説明 | デフォルト |
|----------|------|-----------|
| `name` | サーバーの識別名 | 必須 |
| `runtimeExecutable` | 実行コマンド（`npm`、`yarn`、`node` など） | 必須 |
| `runtimeArgs` | コマンドへの引数 | `[]` |
| `port` | サーバーのポート番号 | `3000` |
| `cwd` | プロジェクトルートからの相対作業ディレクトリ | `.` |
| `env` | 追加の環境変数 | `{}` |
| `autoPort` | ポート競合時の対処（true: 自動、false: エラー） | `true` |

## 5. ビジュアルdiff レビュー

Claude がコードを変更すると、変更行数のインジケーター（例: `+12 -1`）が会話中に表示されます。

### diff レビューの手順

1. **インジケーターをクリック** → diff ビューアが開く
2. **左側**: 変更のあったファイルリスト
3. **右側**: 各ファイルの変更内容（追加は緑、削除は赤）
4. **特定の行をクリック** → コメントを追加できる
5. **複数行にコメント後**: `Cmd+Enter`（Mac）/ `Ctrl+Enter`（Win）で一括送信
6. Claude がコメントを読んで変更を加え、新しい diff が表示される

この「変更 → レビュー → コメント → 修正」のサイクルを繰り返すことで、段階的にコードを仕上げていけます。

## 6. PR 監視

プルリクエストを作成すると、セッション内に CI ステータスバーが表示されます。

### 自動修正（Auto-fix）

CI が失敗すると、Claude が自動的に失敗原因を分析し、修正を試みます。テストの修正、型エラーの解消、リントエラーの修正などを自動で行います。

### 自動マージ（Auto-merge）

すべての CI チェックが通ると、PR を自動的にマージします（スカッシュ方式）。

### 前提条件

| 要件 | 説明 |
|-----|------|
| **GitHub CLI** | `gh` がインストールされ、認証済みであること |
| **リポジトリ設定** | 自動マージがリポジトリの設定で有効化されていること |
| **ブランチ保護ルール** | マージに必要なチェックが設定されていること |

### PR 監視の活用フロー

```
1. Claude にコード変更を依頼
2. diff レビューで確認
3. Claude に PR を作成させる
4. CI ステータスバーが表示される
5. CI 失敗 → Claude が自動修正を試みる
6. 全チェック通過 → 自動マージ（設定済みの場合）
```

この一連のフローにより、PR 作成から CI 修正、マージまでを Claude に任せることができます。

## 7. ビジュアルdiff レビューの詳細なワークフロー

ビジュアルdiff レビューは、デスクトップアプリの大きな強みの一つです。CLI のテキストベース diff とは異なり、グラフィカルな UI で直感的にコードレビューを行えます。

### レビューのベストプラクティス

1. **まず全体を俯瞰する**: diff ビューア左側のファイルリストで、どのファイルが変更されたかを確認する
2. **重要なファイルから確認する**: ビジネスロジックを含むファイルを優先的にレビューする
3. **コメントは具体的に**: 「ここを修正して」ではなく「この条件分岐に null チェックを追加して」のように具体的に書く
4. **一括送信を活用する**: 複数のコメントを追加してから `Cmd+Enter` / `Ctrl+Enter` で一括送信する

### diff レビューとプランモードの組み合わせ

効果的なワークフロー：

```
1. Plan mode で Claude に計画を提示させる
2. 計画をレビューし、必要に応じて修正指示を出す
3. Ask permissions または Auto accept edits に切り替えて実行
4. diff ビューアで変更をレビュー
5. コメントで追加修正を指示
6. 満足するまで 4-5 を繰り返す
```

## 8. コネクタと CLI からのハンドオフ

### 外部ツールのコネクタ

ローカルおよび SSH セッションでは、プロンプトボックスの **+** ボタン → **Connectors** から外部サービスを接続できます：

- Google Calendar、Slack、GitHub、Linear、Notion など

コネクタを接続すると、Claude がカレンダーの読み取り、メッセージ送信、Issue 作成などを直接行えます。

### /desktop コマンド（CLI からのハンドオフ）

ターミナルの CLI セッションをデスクトップアプリに移行できます：

```bash
/desktop
```

このコマンドを実行すると、セッションが保存されてデスクトップアプリで開き、CLI は終了します。

> **公式ドキュメントより**: `/desktop` コマンドは macOS と Windows のみ対応です。

### /desktop コマンドの活用シーン

`/desktop` は以下のような場面で特に便利です：

- CLI で作業を進めていたが、ビジュアルdiff でレビューしたくなった場合
- CLI で立てた計画をデスクトップアプリのアプリプレビューで確認したい場合
- CLI での作業をデスクトップアプリの PR 監視に引き継ぎたい場合

### CLI との比較

| 機能 | CLI | デスクトップ |
|-----|-----|------------|
| 全パーミッションモード | `dontAsk` を含む全モード | Ask / Auto accept / Plan / Bypass |
| サードパーティプロバイダー | 対応 | 非対応（Anthropic API 直接接続のみ） |
| MCP サーバー設定 | 設定ファイル | コネクタ UI または設定ファイル |
| ファイル添付 | 非対応 | 画像・PDF 対応 |
| セッション分離 | `--worktree` フラグ | 自動 worktree |
| スクリプト・自動化 | `--print` フラグ等 | 非対応 |
| アプリプレビュー | 非対応 | 対応 |
| PR 監視 | 非対応 | 対応（自動修正・自動マージ） |

## ハンズオン演習

### 演習 1: デスクトップアプリでセッションを開始する

**目的**: デスクトップアプリのインストール、セッションの開始、パーミッションモードの切り替えを体験する

**前提条件**: Claude Code デスクトップアプリがインストール済み

**手順**:

1. デスクトップアプリを起動する
2. 環境として「ローカル」を選択する
3. プロジェクトフォルダとして、既存のプロジェクトディレクトリを選択する
4. パーミッションモードを「**Plan mode**」に設定する
5. 「このプロジェクトのアーキテクチャを分析して、改善点を提案してください」と入力して送信する
6. Claude の計画（Plan）を確認する
7. パーミッションモードを「**Ask permissions**」に切り替えて、計画の一部を実行してみる

**期待される結果**: Plan mode で変更なしに計画が提示され、Ask permissions に切り替えると各変更前に承認を求められること

### 演習 2: 並列セッションの体験

**目的**: Git worktree による並列セッション分離を体験する

**前提条件**: Git リポジトリがプロジェクトとして設定されていること

**手順**:

1. 1つ目のセッションで「README.md にプロジェクトの説明を追加して」と依頼する
2. サイドバーの **+ New session** をクリックして2つ目のセッションを作成する
3. 2つ目のセッションで「テストファイルを追加して」と依頼する
4. 両方のセッションが独立して動作していることを確認する
5. 各セッションがどのブランチで作業しているか確認する
6. 完了後、不要なセッションをサイドバーからアーカイブする

**期待される結果**: 2つのセッションが独立したブランチで並行作業し、互いの変更が干渉しないこと

### 演習 3: アプリプレビューと diff レビュー

**目的**: アプリプレビューと diff レビューの連携を体験する

**前提条件**: フロントエンドプロジェクト（React、Vue、Next.js 等）が用意されていること

**手順**:

1. `.claude/launch.json` にサーバー設定を記述する
2. Claude に「ヘッダーの色を青に変更して」と依頼する
3. diff インジケーター（`+12 -1` 等）をクリックして diff ビューアを開く
4. 変更行にコメントを追加する（例: 「もう少し暗い青にして」）
5. `Cmd+Enter` でコメントを送信する
6. Claude の修正を確認し、アプリプレビューで反映を確認する

**期待される結果**: diff ビューアでコメントベースのレビューができ、アプリプレビューで変更が反映されること

## よくある質問

**Q: デスクトップアプリと CLI はどちらを使うべきですか？**
A: ビジュアルdiff レビュー、アプリプレビュー、PR 監視が必要な場合はデスクトップアプリが適しています。スクリプトや自動化、サードパーティプロバイダーを使う場合は CLI が適しています。`/desktop` コマンドで CLI からデスクトップアプリへのハンドオフもできます。

**Q: サードパーティプロバイダー（Bedrock、Vertex AI）は使えますか？**
A: デスクトップアプリは Anthropic API に直接接続するため、サードパーティプロバイダーには非対応です。Bedrock や Vertex AI を使う場合は CLI を使用してください。

**Q: Worktree が不要な場合はどうしますか？**
A: セッション完了後、サイドバーでセッションにホバーしてアーカイブアイコンをクリックすると、worktree が削除されます。カスタムの worktree 保存場所は設定で変更できます。

**Q: PR 監視が動作しません**
A: GitHub CLI（`gh`）がインストール・認証済みか確認してください。また、リポジトリの設定で自動マージが有効になっている必要があります。

**Q: コネクタの設定はどこに保存されますか？**
A: コネクタの設定はデスクトップアプリ内で管理されます。CLI の `~/.claude/settings.json` とは別に管理されるため、CLI から直接コネクタを操作することはできません。

**Q: launch.json の autoPort とは何ですか？**
A: `autoPort` が `true`（デフォルト）の場合、指定したポートが既に使用中であれば別のポートに自動切り替えします。`false` の場合はエラーになります。複数のプロジェクトを同時に開発している場合は `true` が便利です。

## まとめ

この章で学んだ重要ポイント：

- デスクトップアプリは macOS（Intel/Apple Silicon）と Windows（x64）に対応
- パーミッションモードは Plan mode で計画確認後に Ask permissions で実行するのがベストプラクティス
- 並列セッションは Git worktree で自動分離され、各セッションが独立したブランチで作業する
- アプリプレビューの autoVerify で変更後の自動検証が可能
- diff レビューのインラインコメントで段階的にコードを仕上げられる
- PR 監視で CI 失敗の自動修正と全チェック通過後の自動マージが可能
