## 1. サブエージェントとは

サブエージェントは、メインのClaude Codeセッションから生成される**特化型のAIアシスタント**です。「特化型」というのがポイントで、それぞれが以下の特性を持って独立に動作します。

- **独自のコンテキストウィンドウ**: メインの会話コンテキストを消費しない
- **カスタムプロンプト**: 専門的な役割に特化したシステムプロンプト
- **ツール制限**: 使えるツールを明示的に制限できる
- **独立したパーミッション**: メインセッションとは別のパーミッション設定

タスクがサブエージェントの説明（description）に合致すると、Claude Codeは自動的にそのサブエージェントへ作業を委譲し、完了後に結果をメインセッションに返します。

### サブエージェントを使う理由

| メリット | 説明 |
|---------|------|
| **コンテキスト保全** | 探索や実装作業をメイン会話から切り離し、メインのコンテキストを節約する |
| **制約の強制** | 特定のツールしか使えないよう制限し、安全性を高める |
| **設定の再利用** | ユーザーレベルで定義すれば、すべてのプロジェクトで共有できる |
| **専門化** | ドメイン特化のシステムプロンプトで精度を向上させる |
| **コスト制御** | 簡単なタスクはHaikuなど低コストモデルに委譲してコストを抑える |

> **公式ドキュメントより**: サブエージェント内でさらにサブエージェントを生成することはできません（ネスト不可）。複数エージェントが互いに通信しながら協調作業する場合は Agent Teams を使用してください。

## 2. 組み込みサブエージェント

Claude Codeには6種類の組み込みサブエージェントが含まれており、適切なタイミングで自動的に使用されます。

### Explore

コードベースの検索・分析に特化した高速な読み取り専用エージェントです。

```
モデル:    Haiku（高速・低レイテンシ・低コスト）
ツール:    読み取り専用ツール（Write/Editは使用不可）
用途:      ファイル検索、コード検索、コードベース調査
深さ設定:  quick / medium / very thorough
```

Claude Codeがコードベースを理解する必要があるが変更は不要な場合に自動的に使用されます。調査結果をメインのコンテキストに持ち込まずに済むのが最大の利点です。

### Plan

プランモード中にコンテキストを収集するための調査エージェントです。

```
モデル:    メイン会話から継承
ツール:    読み取り専用ツール
用途:      計画策定のためのコードベース調査
```

プランモード（Shift+Tab）では、メインセッション自体がサブエージェントを生成できないため、このPlanエージェントが代わりにコードベースを調査します。

### general-purpose

探索と実装の両方が必要な複雑・多ステップタスク向けの汎用エージェントです。

```
モデル:    メイン会話から継承
ツール:    すべてのツール
用途:      複雑な調査、複数ステップの操作、コード変更
```

### Bash

ターミナルコマンドを別コンテキストで実行するエージェントです。

```
モデル:    メイン会話から継承
ツール:    ターミナルコマンド
用途:      別コンテキストでのシェルコマンド実行
```

### statusline-setup

`/statusline`コマンドの設定を支援する専用エージェントです。

```
モデル:    Sonnet
用途:      /statusline 設定のガイド
```

### Claude Code Guide

Claude Codeの機能に関する質問に回答するヘルプエージェントです。

```
モデル:    Haiku
用途:      Claude Code機能の質問に回答
```

## 3. カスタムサブエージェントの作成

### `/agents`コマンドによる対話型作成

最も手軽な方法は`/agents`コマンドです。

```
/agents
```

このコマンドで以下の操作が対話的に行えます：
- サブエージェントの一覧表示（組み込み・ユーザー・プロジェクト・プラグイン）
- 新規作成（ガイド付きセットアップ or Claude生成）
- 既存エージェントの編集・削除

### ファイルによる手動定義

サブエージェントは**YAMLフロントマター + Markdownプロンプト**の形式で定義します。

#### 配置場所と優先順位

スコープの優先順位は明確に定められています。同名のサブエージェントが複数存在する場合、番号の小さい方が優先されます。

| 優先度 | 場所 | スコープ |
|--------|------|---------|
| 1（最高） | `--agents` CLIフラグ | 現在のセッションのみ |
| 2 | `.claude/agents/` | 現在のプロジェクト |
| 3 | `~/.claude/agents/` | すべてのプロジェクト |
| 4（最低） | プラグインの`agents/`ディレクトリ | プラグインが有効な場所 |

#### 基本的なファイル形式

```markdown
---
name: code-reviewer
description: コードの品質とベストプラクティスをレビューする
tools: Read, Glob, Grep
model: sonnet
---

あなたはシニアコードレビュアーです。コードを分析し、品質・セキュリティ・ベストプラクティスについて
具体的でアクションに移せるフィードバックを提供してください。
```

YAMLフロントマター（`---`で囲まれた部分）が設定、その下のMarkdownがシステムプロンプトです。

### フロントマターの全設定フィールド

| フィールド | 必須 | 説明 |
|-----------|------|------|
| `name` | **必須** | 小文字とハイフンのみの一意な識別子 |
| `description` | **必須** | Claudeがいつこのサブエージェントに委譲すべきかの判断基準 |
| `tools` | 任意 | 使用可能なツールの許可リスト（省略時は全ツール継承） |
| `disallowedTools` | 任意 | 拒否するツールのリスト |
| `model` | 任意 | `sonnet`/`opus`/`haiku`/`inherit`（省略時は`inherit`） |
| `permissionMode` | 任意 | `default`/`acceptEdits`/`dontAsk`/`bypassPermissions`/`plan` |
| `maxTurns` | 任意 | 最大ターン数（超えると停止） |
| `skills` | 任意 | 起動時に注入するスキルのリスト |
| `mcpServers` | 任意 | このサブエージェントで利用可能なMCPサーバー |
| `hooks` | 任意 | このサブエージェント専用のライフサイクルフック |
| `memory` | 任意 | 永続メモリスコープ: `user`/`project`/`local` |
| `background` | 任意 | `true`にすると常にバックグラウンド実行 |
| `isolation` | 任意 | `worktree`にするとgit worktreeで分離実行 |

### `--agents`フラグによる動的定義

セッション開始時にCLIフラグでサブエージェントを定義することもできます。

```bash
claude --agents '{
  "code-reviewer": {
    "description": "コード変更後に積極的にレビューする専門家。",
    "prompt": "あなたはシニアコードレビュアーです。コード品質、セキュリティ、ベストプラクティスに集中してください。",
    "tools": ["Read", "Grep", "Glob", "Bash"],
    "model": "sonnet"
  }
}'
```

このフラグはセッション限定で、ファイルには保存されません。テストや自動化スクリプトに便利です。

### Agent spawning制限

`tools`フィールドでサブエージェントが生成できるサブエージェントの種類を制限できます。

```yaml
tools: Agent(worker, researcher), Read, Bash
```

この例では`worker`と`researcher`という名前のサブエージェントのみ使用可能です。

## 4. 高度な設定

### モデルの選択

タスクの複雑さとコストのバランスを考慮して適切なモデルを選びます。

```yaml
model: sonnet   # Sonnet 4.6 - コーディング全般に最適
model: opus     # Opus 4.6 - 複雑な推論・設計判断
model: haiku    # Haiku 4.5 - 単純な読み取り・検索タスク
model: inherit  # メイン会話と同じモデル（デフォルト）
```

### ツール制限

許可リストと拒否リストの両方で制御できます。

```yaml
---
name: safe-researcher
description: 読み取り専用の安全な調査エージェント
tools: Read, Grep, Glob, Bash
disallowedTools: Write, Edit
---
```

### スキルのプリロード

起動時からドメイン知識を持たせることができます。

```yaml
---
name: api-developer
description: チームの規約に従ってAPIエンドポイントを実装する
skills:
  - api-conventions
  - error-handling-patterns
---

APIエンドポイントを実装してください。プリロードされたスキルの規約とパターンに従ってください。
```

### 永続メモリ

サブエージェントに会話をまたいで学習・記憶させることができます。

```yaml
---
name: code-reviewer
description: コードの品質とベストプラクティスをレビューする
memory: user
---

あなたはコードレビュアーです。コードをレビューしながら、発見したパターンや規約、
繰り返し発生する問題をエージェントメモリに記録してください。
```

| スコープ | 保存場所 | 用途 |
|---------|---------|------|
| `user` | `~/.claude/agent-memory/<名前>/` | すべてのプロジェクト共通の学習 |
| `project` | `.claude/agent-memory/<名前>/` | プロジェクト固有・バージョン管理可能 |
| `local` | `.claude/agent-memory-local/<名前>/` | プロジェクト固有・ローカルのみ |

### フックによる条件制御

`hooks`フィールドでツール使用前の検証を設定できます。例えば読み取り専用DBクエリのみ許可するサブエージェント：

```yaml
---
name: db-reader
description: 読み取り専用のデータベースクエリを実行する
tools: Bash
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly-query.sh"
---
```

検証スクリプト例（SQLの書き込み操作をブロック）：

```bash
#!/bin/bash
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

if echo "$COMMAND" | grep -iE '\b(INSERT|UPDATE|DELETE|DROP|CREATE|ALTER|TRUNCATE)\b' > /dev/null; then
  echo "ブロック: SELECT クエリのみ許可されています" >&2
  exit 2
fi

exit 0
```

プロジェクト設定ファイルでサブエージェントのライフサイクルを制御するフック（`SubagentStart`/`SubagentStop`）も利用可能です。

### フォアグラウンドとバックグラウンド

サブエージェントはフォアグラウンド（デフォルト）とバックグラウンドの2つのモードで動作します。

- **フォアグラウンド**: パーミッションプロンプトや質問がメインセッションに渡される
- **バックグラウンド**: メインセッションと並行して動作。`Ctrl+B`で切り替え可能

```yaml
# 常にバックグラウンドで実行する設定
background: true
```

バックグラウンドを完全に無効化するには環境変数を設定します：

```bash
CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1
```

### 自動コンパクション

サブエージェントのコンテキスト使用率がデフォルトで約95%に達すると自動的にコンパクションが実行されます。この閾値は環境変数で変更可能です：

```bash
CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=80
```

### 特定のサブエージェントを無効化

`settings.json`で特定のサブエージェントの使用を禁止できます：

```json
{
  "permissions": {
    "deny": ["Agent(Explore)", "Agent(my-custom-agent)"]
  }
}
```

## 5. 実践例: サブエージェントの設計パターン

### コードレビュアー

```markdown
---
name: code-reviewer
description: コードの品質・セキュリティ・保守性を専門的にレビューする。コード変更後に積極的に使用する。
tools: Read, Grep, Glob, Bash
model: inherit
---

あなたはシニアコードレビュアーです。高い品質基準を維持します。

起動時の手順:
1. git diff で最近の変更を確認
2. 変更されたファイルに集中
3. すぐにレビューを開始

レビューチェックリスト:
- コードは明確で読みやすいか
- 関数名・変数名は適切か
- コードの重複はないか
- 適切なエラーハンドリングがあるか
- シークレットやAPIキーが露出していないか
- 入力バリデーションが実装されているか
- テストカバレッジは十分か

優先度別にフィードバックを提供する:
- 重大な問題（必ず修正）
- 警告（修正すべき）
- 提案（改善を検討）
```

### デバッガー

```markdown
---
name: debugger
description: エラー・テスト失敗・予期しない動作のデバッグ専門家。問題に遭遇したら積極的に使用する。
tools: Read, Edit, Bash, Grep, Glob
---

あなたは根本原因分析の専門家です。

デバッグプロセス:
1. エラーメッセージとスタックトレースを取得
2. 再現手順を特定
3. 仮説を3つ以上リストアップ
4. 仮説を順に検証
5. 最小限の修正を実装
6. 解決策を検証

各問題について提供する情報:
- 根本原因の説明
- 診断を裏付ける証拠
- 具体的なコード修正
- テストアプローチ
- 防止策の推奨
```

## ハンズオン演習

### 演習 1: コードレビュアーサブエージェントの作成

**目的**: カスタムサブエージェントを定義し、自動的に委譲されることを確認する
**前提条件**: Claude Codeがインストール済みで、Gitリポジトリ内で作業していること

**手順**:
1. プロジェクトルートに`.claude/agents/`ディレクトリを作成する
   ```bash
   mkdir -p .claude/agents
   ```
2. `code-reviewer.md`を以下の内容で作成する
   ```markdown
   ---
   name: code-reviewer
   description: コード変更のレビューを行う。git diffの確認やコード品質チェックに使用する。
   tools: Read, Grep, Glob, Bash
   model: sonnet
   ---

   あなたはコードレビュアーです。変更されたコードを分析し、
   品質・セキュリティ・ベストプラクティスの観点でフィードバックを提供してください。
   ```
3. Claude Codeを起動し、何かファイルを変更した後に以下のように依頼する
   ```
   最近の変更をレビューしてください
   ```
4. サブエージェントが自動的に呼び出されることを確認する

**期待される結果**: Claude Codeがcode-reviewerサブエージェントに自動的に委譲し、変更内容に対するレビューフィードバックが返される

### 演習 2: 読み取り専用調査エージェントの作成

**目的**: ツール制限を使って安全な調査専用エージェントを作成する

**手順**:
1. `~/.claude/agents/researcher.md`を以下の内容で作成する
   ```markdown
   ---
   name: researcher
   description: コードベースの調査・分析を行う読み取り専用エージェント
   tools: Read, Grep, Glob
   disallowedTools: Write, Edit, Bash
   model: haiku
   ---

   あなたは読み取り専用の調査員です。コードベースを分析し、
   構造・パターン・依存関係について報告してください。
   コードの変更は一切行わないでください。
   ```
2. Claude Codeで以下のように使用する
   ```
   このプロジェクトの認証フローを調査して概要を報告してください
   ```
3. researcherサブエージェントが読み取り専用のツールだけで調査を行うことを確認する

**期待される結果**: Haikuモデルで高速に調査が完了し、メインのコンテキストを消費せずに結果が返される

### 演習 3: `--agents`フラグでの動的定義テスト

**目的**: CLIフラグによるセッション限定のサブエージェント定義を試す

**手順**:
1. 以下のコマンドでClaude Codeを起動する
   ```bash
   claude --agents '{
     "test-helper": {
       "description": "テストの作成と実行を支援する",
       "prompt": "あなたはテストの専門家です。テストの作成、実行、失敗の修正を支援してください。",
       "tools": ["Read", "Edit", "Bash", "Grep"],
       "model": "sonnet"
     }
   }'
   ```
2. テストに関する依頼をして、test-helperサブエージェントが使われることを確認する
3. セッション終了後に`/agents`で確認し、動的に定義したエージェントが消えていることを確認する

**期待される結果**: セッション中のみtest-helperサブエージェントが利用可能で、セッション終了後は消える

## よくある質問

**Q: サブエージェントとAgent Teamsはどう使い分ければよいですか？**
A: サブエージェントは「結果だけが欲しい一方向のタスク」に最適です。調査結果をメインに返す、レビュー結果を報告する、など。Agent Teamsは「メンバー同士が議論・協力する必要がある複雑な作業」に使います。トークンコストはサブエージェントの方が大幅に低いため、まずはサブエージェントで対応できないか検討してください。

**Q: サブエージェントがネストできないのはなぜですか？**
A: 無限ループやコスト爆発を防ぐための設計上の制約です。サブエージェントがさらにサブエージェントを生成すると、制御が困難になりトークン消費が指数的に増加するリスクがあります。

**Q: `description`フィールドはどのくらい重要ですか？**
A: 非常に重要です。Claude Codeはこのdescriptionを見て「このタスクをこのサブエージェントに委譲すべきか」を判断します。曖昧な記述だと意図しないタイミングで呼ばれたり、逆に必要な時に呼ばれなかったりします。具体的に「いつ使うべきか」を明記してください。

**Q: サブエージェントのメモリはどのように管理されますか？**
A: `memory`フィールドを設定すると、サブエージェントは`~/.claude/agent-memory/<名前>/`（userスコープの場合）にメモリファイルを保存します。セッションを跨いで学習内容が蓄積されるため、繰り返し使うサブエージェント（レビュアーなど）で特に有効です。

**Q: Auto-compactionの閾値はどう調整すればよいですか？**
A: デフォルトでは約95%でコンパクションが実行されます。サブエージェントが大量のコンテキストを必要とする場合は`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`で閾値を下げることを検討してください。例えば`80`に設定すると、80%の時点でコンパクションが発生します。

## まとめ

この章で学んだ重要ポイント：

- サブエージェントは独立したコンテキストウィンドウ・カスタムプロンプト・ツール制限・独立パーミッションを持つ特化型AI
- 組み込みサブエージェントは6種類（Explore、Plan、general-purpose、Bash、statusline-setup、Claude Code Guide）で、適切なタイミングで自動呼び出しされる
- カスタムサブエージェントは`.claude/agents/`（プロジェクト）または`~/.claude/agents/`（ユーザー）にYAMLフロントマター付きMarkdownで定義する
- スコープ優先順位は `--agents`フラグ > `.claude/agents/` > `~/.claude/agents/` > プラグイン
- `memory`フィールドで永続メモリ、`hooks`フィールドで条件制御、`skills`フィールドでドメイン知識のプリロードが可能
