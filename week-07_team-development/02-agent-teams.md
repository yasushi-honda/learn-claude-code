# Agent Teamsによる並列処理

> 対応する公式ドキュメント: [Orchestrate teams of Claude Code sessions](https://code.claude.com/docs/en/agent-teams)

## 学習目標

- Agent Teamsの概要とサブエージェントとの違いを理解する
- チームリーダーとメンバーの役割を把握する
- TaskCreate・SendMessage等のツールによるタスク管理・メッセージングを理解する
- `--teammate-mode`設定（auto/in-process/tmux）の使い方を学ぶ
- Agent Teamsが有効な場面と適切なチーム構成パターンを習得する

## 概要

Agent Teamsは、複数のClaude Codeインスタンスが協調して作業するための機能です。1つのセッションがチームリーダーとなり、作業の調整、タスクの割り当て、結果の統合を担当します。メンバーはそれぞれ独立したコンテキストウィンドウを持ち、直接互いにコミュニケーションを取れます。

サブエージェントとの大きな違いは、**メンバー同士が直接通信できる**点です。共有タスクリストを使って自律的に作業を調整することもできます。

> **注意**: Agent Teamsはデフォルトで無効です。`settings.json`または環境変数で有効化する必要があります。また、実験的な機能のため、セッション再開やタスク調整に既知の制限があります。

## 本文

### Agent Teamsの有効化

```json
// settings.json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

または環境変数として：

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

### サブエージェントとAgent Teamsの比較

| 特性 | サブエージェント | Agent Teams |
|------|----------------|-------------|
| **コンテキスト** | 独自のコンテキストウィンドウ、結果を呼び出し元に返す | 独自のコンテキストウィンドウ、完全に独立 |
| **通信** | メインエージェントにのみ結果を報告 | メンバー同士が直接メッセージ |
| **調整** | メインエージェントが全作業を管理 | 共有タスクリストで自己調整 |
| **最適な用途** | 結果だけが重要な集中的タスク | 議論・協力が必要な複雑な作業 |
| **トークンコスト** | 低い（結果がメインコンテキストに要約される） | 高い（各メンバーが独立したClaudeインスタンス） |

### アーキテクチャ

```
チームリード（メインセッション）
    ├── タスクリスト（共有）
    ├── メールボックス（メッセージシステム）
    ├── メンバー1（独立したClaudeインスタンス）
    ├── メンバー2（独立したClaudeインスタンス）
    └── メンバー3（独立したClaudeインスタンス）
```

**コンポーネント：**

| コンポーネント | 役割 |
|--------------|------|
| **チームリード** | チームを作成し、メンバーを生成し、作業を調整するメインセッション |
| **メンバー** | 割り当てられたタスクに取り組む独立したClaudeインスタンス |
| **タスクリスト** | メンバーが要求・完了する共有作業リスト |
| **メールボックス** | エージェント間のメッセージングシステム |

チームの設定とタスクはローカルに保存されます：
- **チーム設定**: `~/.claude/teams/{team-name}/config.json`
- **タスクリスト**: `~/.claude/tasks/{team-name}/`

### チームの開始

Agent Teamsを有効にした後、自然言語でチームの作成を指示します：

```
CLIツールのTODOコメント追跡機能を設計したい。
異なる観点から探索するAgent Teamを作成してください：
UXに取り組むメンバー1人、技術アーキテクチャに取り組むメンバー1人、
悪魔の代弁者となるメンバー1人。
```

Claudeはチームを作成し、各メンバーを生成し、問題を探索させ、発見事項を統合します。

### 表示モードの選択

Agent Teamsは2つの表示モードをサポートしています：

**in-process（デフォルト）:**
- すべてのメンバーがメインターミナル内で動作
- `Shift+Down`でメンバーを切り替え、直接メッセージを送信
- どんなターミナルでも動作、追加セットアップ不要

**split-panes（分割ペイン）:**
- 各メンバーが独自のペインを持つ
- 全員の出力を同時確認、ペインをクリックして直接操作
- tmuxまたはiTerm2が必要

```json
// settings.json
{
  "teammateMode": "in-process"
}
```

```bash
# セッション単位での指定
claude --teammate-mode in-process
```

デフォルトは`"auto"`で、tmuxセッション内であれば分割ペイン、そうでなければin-processを使用します。

### タスクの管理

共有タスクリストでチームの作業を調整します：

**タスクの状態：**
- `pending`（未着手）
- `in_progress`（実行中）
- `completed`（完了）

**タスクの割り当て方法：**

1. **リードが割り当て**: 特定のタスクを特定のメンバーに指示
2. **自己要求（self-claim）**: タスク完了後、メンバーが次の未割り当てタスクを自発的に取得

タスクの要求はファイルロックを使用してレースコンディションを防ぎます。

```
# タスクの依存関係
タスクBとCはタスクAが完了するまでブロックされます。
タスクAが完了すると、依存するタスクは自動的にブロック解除されます。
```

### メッセージングシステム

メンバー間は以下のメッセージタイプで通信します：

**message（ダイレクトメッセージ）:**
```
特定のメンバーにメッセージを送信します。
受信者を指定する必要があります。
```

**broadcast（全員へのブロードキャスト）:**
```
すべてのメンバーに同じメッセージを一斉送信します。
コストがチームサイズに比例して増加するため、慎重に使用してください。
```

### チームの制御

**メンバーへの直接指示：**
- in-processモード: `Shift+Down`でメンバーを切り替え、入力でメッセージ送信
- 分割ペインモード: ペインをクリックして直接操作

**メンバーのシャットダウン：**
```
「researcherメンバーをシャットダウンしてください」
```
リードがシャットダウンリクエストを送信。メンバーは承認（終了）または拒否（継続）できます。

**チームのクリーンアップ：**
```
「チームをクリーンアップしてください」
```
共有リソースを削除します。アクティブなメンバーが残っている場合は失敗します。

> **重要**: クリーンアップは必ずリードが実行してください。メンバーがクリーンアップを実行すると、チームコンテキストが正しく解決されず、リソースが不整合な状態になる可能性があります。

### 計画承認の要求

複雑またはリスクの高いタスクでは、実装前にメンバーの計画を承認させることができます：

```
「認証モジュールをリファクタリングするarchitectメンバーを生成してください。
変更を加える前に計画の承認を要求してください。」
```

承認フロー：
1. メンバーが読み取り専用のプランモードで計画を策定
2. リードに計画承認リクエストを送信
3. リードがレビューして承認またはフィードバック付きで拒否
4. 拒否された場合、メンバーはプランモードのまま修正して再送
5. 承認後、メンバーはプランモードを終了して実装開始

### Agent Teamsの活用パターン

#### パターン1: 並列コードレビュー

```
PR #142のレビューにAgent Teamを使用してください。3人のレビュアーを生成してください：
- セキュリティへの影響に集中する1人
- パフォーマンスへの影響を確認する1人
- テストカバレッジを検証する1人
それぞれにレビューと所見の報告を行わせてください。
```

各レビュアーは同じPRを異なる視点でレビューし、リードが全員の所見を統合します。

#### パターン2: 競合する仮説での調査

```
ユーザーから1つのメッセージ後にアプリが切断されるとの報告があります。
5人のAgent Teammateを生成して異なる仮説を調査させてください。
互いの理論を反証し合うよう指示してください。
```

複数の独立した調査者が積極的に互いの理論を反証することで、単一の調査では見逃しがちな根本原因を特定できます。

#### パターン3: 並列実装

フロントエンド・バックエンド・テストをそれぞれ独立したメンバーが担当し、異なるファイルセットを所有することで競合なく並列実装できます。

### ベストプラクティス

**メンバーへの十分なコンテキスト提供：**
メンバーはCLAUDE.md、MCPサーバー、スキルを自動的に読み込みますが、リードの会話履歴は継承しません。タスク固有の詳細をスポーンプロンプトに含めてください。

```
「src/auth/のセキュリティ脆弱性をレビューするsecurity-reviewerメンバーを生成してください。
プロンプト: 認証モジュールをセキュリティの観点でレビューしてください。
トークン処理、セッション管理、入力バリデーションに集中してください。
JWTトークンをhttpOnly cookieに保存しています。重大度評価付きで問題を報告してください。」
```

**適切なチームサイズ：**
- 実用的なワークフローでは3〜5人のメンバーが適切
- メンバー当たり5〜6タスクが生産性を維持する目安
- 5〜6人以上になると調整オーバーヘッドが増大

**ファイル競合を避ける：**
同じファイルを2人のメンバーが編集すると上書きが発生します。各メンバーが異なるファイルセットを担当するように作業を分割してください。

**タスクの適切なサイズ設定：**
- 小さすぎる: 調整オーバーヘッドがメリットを上回る
- 大きすぎる: メンバーが長時間チェックインなしで作業し、無駄なコストリスクが増える
- 適切なサイズ: 明確な成果物を生む自己完結した単位（関数・テストファイル・レビューなど）

### トークンコストへの注意

Agent Teamsはシングルセッションと比べて大幅にトークンを消費します。各メンバーが独立したコンテキストウィンドウを持つため、使用量はアクティブなメンバー数に比例します。

コストを管理するためのヒント：
- メンバーにはSonnetを使用（能力とコストのバランス）
- チームをできるだけ小さく保つ
- スポーンプロンプトを絞り込んで初期コンテキストを削減
- 作業が完了したらすぐにチームをクリーンアップ

### 制限事項

Agent Teamsは実験的な機能です。現在の既知の制限：

- **in-processメンバーのセッション再開不可**: `/resume`と`/rewind`はin-processメンバーを復元しない
- **タスク状態の遅延**: メンバーがタスクを完了済みにしない場合がある
- **シャットダウンの遅延**: メンバーは現在のリクエストやツール呼び出しを完了してからシャットダウン
- **1セッション1チーム**: リードは一度に1チームのみ管理可能
- **ネストなし**: メンバーはさらにチームやメンバーを生成できない
- **固定リード**: チームを作成したセッションがそのチームのリードで変更不可
- **分割ペインはtmux/iTerm2が必要**: VS Code統合ターミナル、Windows Terminal、Ghosttyでは非対応

## まとめ

- Agent Teamsは複数のClaude Codeインスタンスが協調する実験的機能（デフォルト無効）
- サブエージェントと異なり、メンバー同士が直接通信し共有タスクリストで自律的に調整できる
- チームリード、メンバー、タスクリスト、メールボックスで構成される
- in-processとsplit-panesの2つの表示モードがある
- 並列コードレビュー・競合する仮説の調査・大規模リファクタリングなどに最適
- トークンコストはシングルセッションより大幅に高くなるため、本当に並列化が必要な場合のみ使用する
- ファイル競合を避けるため、各メンバーが異なるファイルセットを担当するよう設計する
