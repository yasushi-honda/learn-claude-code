# Agent Teamsによる並列処理

> **対応公式ドキュメント**: https://code.claude.com/docs/en/agent-teams
> **想定所要時間**: 約60分
> **難易度**: ★★★★☆
> **公式ドキュメント検証日**: 2026-03-26

## この章の学習目標

この章を終えると、以下のことができるようになります：

1. Agent Teamsの仕組みとサブエージェントとの違いを明確に説明できる
2. チームリーダー・メンバー・タスクリスト・メールボックスの各コンポーネントの役割を理解できる
3. in-processモードとsplit-panesモードを使い分けられる
4. 適切なチーム構成（3-5人、5-6タスク/人）で並列作業を設計できる

---

## 1. Agent Teamsとは

Agent Teamsは、複数のClaude Codeインスタンスが**協調して作業する**ための実験的機能です。1つのセッションがチームリーダーとなり、作業の調整・タスクの割り当て・結果の統合を担当します。メンバーはそれぞれ独立したコンテキストウィンドウを持ち、**直接互いにコミュニケーション**を取れます。

> **公式ドキュメントより**: Agent Teamsはデフォルトで無効の実験的機能です。`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`で有効化する必要があります。セッション再開やタスク調整に既知の制限があります。

### サブエージェントとの比較

「サブエージェントで十分ではないか？」と思うかもしれません。この比較表で違いを明確にしましょう。

| 特性 | サブエージェント | Agent Teams |
|------|----------------|-------------|
| **通信方向** | メインに結果を返すのみ | メンバー同士が直接メッセージ |
| **調整方法** | メインエージェントが全作業を管理 | 共有タスクリストで自己調整 |
| **コンテキスト** | 結果がメインに要約される | 各メンバーが完全に独立 |
| **最適な用途** | 結果だけが重要な集中的タスク | 議論・協力が必要な複雑な作業 |
| **トークンコスト** | 低い | 高い（各メンバーが独立したインスタンス） |

**判断基準**: まずサブエージェントで対応できないか検討し、メンバー間の協力・議論が本当に必要な場合にのみAgent Teamsを使用してください。

## 2. アーキテクチャ

### 4つのコンポーネント

```
チームリード（メインセッション）
    ├── タスクリスト（共有）
    ├── メールボックス（メッセージシステム）
    ├── メンバー1（独立したClaudeインスタンス）
    ├── メンバー2（独立したClaudeインスタンス）
    └── メンバー3（独立したClaudeインスタンス）
```

| コンポーネント | 役割 |
|--------------|------|
| **チームリード** | チームを作成し、メンバーを生成し、作業を調整するメインセッション |
| **メンバー（Teammates）** | 割り当てられたタスクに取り組む独立したClaudeインスタンス |
| **タスクリスト** | メンバーが要求・完了する共有作業リスト |
| **メールボックス** | エージェント間のメッセージングシステム |

### ストレージ

チームの設定とタスクはローカルに保存されます：

```
~/.claude/teams/{team-name}/config.json   # チーム設定
~/.claude/tasks/{team-name}/               # タスクリスト
```

## 3. セットアップと基本操作

### 有効化

`settings.json`で有効化する方法：

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

または環境変数として：

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

### チームの開始

有効化後、自然言語でチームの作成を指示します：

```
CLIツールのTODOコメント追跡機能を設計したい。
異なる観点から探索するAgent Teamを作成してください：
UXに取り組むメンバー1人、技術アーキテクチャに取り組むメンバー1人、
悪魔の代弁者となるメンバー1人。
```

Claudeはチームを作成し、各メンバーを生成し、問題を探索させ、発見事項を統合します。

### 表示モードの選択

Agent Teamsは2つの表示モードをサポートしています：

**in-process（デフォルト）:**
- すべてのメンバーがメインターミナル内で動作
- `Shift+Down`でメンバーを切り替え、直接メッセージを送信
- どんなターミナルでも動作、追加セットアップ不要

**split-panes（分割ペイン）:**
- 各メンバーが独自のペインを持つ
- 全員の出力を同時確認、ペインをクリックして直接操作
- tmuxまたはiTerm2が必要

```json
// settings.json
{
  "teammateMode": "in-process"
}
```

```bash
# セッション単位での指定
claude --teammate-mode in-process
```

デフォルトは`"auto"`で、tmuxセッション内であれば分割ペイン、そうでなければin-processを使用します。

## 4. タスク管理とコミュニケーション

### タスクの状態管理

タスクは3つの状態を持ちます：

```
pending（未着手） → in_progress（実行中） → completed（完了）
```

**タスクの依存関係**: タスクBがタスクAに依存する場合、タスクAが完了するまでBは自動的にブロックされます。

**タスクの割り当て方法**:
1. **リードが割り当て**: 特定のタスクを特定のメンバーに指示
2. **自己要求（self-claim）**: タスク完了後、メンバーが次の未割り当てタスクを自発的に取得

タスクの要求はファイルロックを使用してレースコンディションを防ぎます。

### メッセージングシステム

| タイプ | 説明 | 注意点 |
|--------|------|--------|
| **message** | 特定のメンバーにダイレクトメッセージ | 受信者を指定する必要がある |
| **broadcast** | 全メンバーに一斉送信 | コストがチームサイズに比例して増加 |

broadcastは便利ですが、全メンバーのコンテキストにメッセージが追加されるため、コストに注意してください。

### Quality Gates

メンバーの作業品質を制御するための仕組みです：

- **TeammateIdle**: メンバーがアイドルになった時のフック。`exit 2`でフィードバックを返し再作業させる
- **TaskCompleted**: タスク完了時のフック。`exit 2`で完了を阻止し修正を要求する

### 計画承認の要求

リスクの高いタスクでは、実装前にメンバーの計画を承認させることができます：

```
認証モジュールをリファクタリングするarchitectメンバーを生成してください。
変更を加える前に計画の承認を要求してください。
```

承認フロー：
1. メンバーが読み取り専用のプランモードで計画を策定
2. リードに計画承認リクエストを送信
3. リードがレビューして承認またはフィードバック付きで拒否
4. 拒否された場合、メンバーはプランモードのまま修正して再送
5. 承認後、メンバーはプランモードを終了して実装開始

### チームの制御

**メンバーへの直接指示：**
- in-processモード: `Shift+Down`でメンバーを切り替え、入力でメッセージ送信
- 分割ペインモード: ペインをクリックして直接操作

**メンバーのシャットダウン：**
```
researcherメンバーをシャットダウンしてください
```
リードがシャットダウンリクエストを送信。メンバーは承認（終了）または拒否（継続）できます。

**チームのクリーンアップ：**
```
チームをクリーンアップしてください
```

> **公式ドキュメントより**: クリーンアップは必ずリードが実行してください。メンバーがクリーンアップを実行すると、チームコンテキストが正しく解決されず、リソースが不整合な状態になる可能性があります。

## 5. 活用パターンとベストプラクティス

### パターン1: 並列コードレビュー

```
PR #142のレビューにAgent Teamを使用してください。3人のレビュアーを生成してください：
- セキュリティへの影響に集中する1人
- パフォーマンスへの影響を確認する1人
- テストカバレッジを検証する1人
それぞれにレビューと所見の報告を行わせてください。
```

各レビュアーは同じPRを異なる視点でレビューし、リードが全員の所見を統合します。

### パターン2: 競合する仮説での調査

```
ユーザーから1つのメッセージ後にアプリが切断されるとの報告があります。
5人のAgent Teammateを生成して異なる仮説を調査させてください。
互いの理論を反証し合うよう指示してください。
```

複数の独立した調査者が積極的に互いの理論を反証することで、単一の調査では見逃しがちな根本原因を特定できます。

### パターン3: 並列実装

フロントエンド・バックエンド・テストをそれぞれ独立したメンバーが担当し、異なるファイルセットを所有することで競合なく並列実装できます。

### ベストプラクティス

**適切なチームサイズ:**
- **3〜5人のメンバー**が実用的なワークフローに適切
- **メンバー当たり5〜6タスク**が生産性を維持する目安
- 5〜6人以上になると調整オーバーヘッドが増大

**メンバーへの十分なコンテキスト提供:**

メンバーはCLAUDE.md、MCPサーバー、スキルを自動的に読み込みますが、リードの会話履歴は継承しません。タスク固有の詳細をスポーンプロンプトに含めてください。

```
src/auth/のセキュリティ脆弱性をレビューするsecurity-reviewerメンバーを生成してください。
プロンプト: 認証モジュールをセキュリティの観点でレビューしてください。
トークン処理、セッション管理、入力バリデーションに集中してください。
JWTトークンをhttpOnly cookieに保存しています。重大度評価付きで問題を報告してください。
```

**ファイル競合を避ける:**

同じファイルを2人のメンバーが編集すると上書きが発生します。各メンバーが異なるファイルセットを担当するように作業を分割してください。

**タスクの適切なサイズ設定:**
- 小さすぎる: 調整オーバーヘッドがメリットを上回る
- 大きすぎる: メンバーが長時間チェックインなしで作業し、無駄なコストリスクが増える
- 適切なサイズ: 明確な成果物を生む自己完結した単位（関数・テストファイル・レビューなど）

### トークンコストへの注意

Agent Teamsはシングルセッションの**約7倍**のトークンを消費することがあります（メンバーがプランモードで実行する場合）。各メンバーが独立したコンテキストウィンドウを持つため、使用量はアクティブなメンバー数に比例します。

コスト管理のポイント：
- メンバーにはSonnetを使用（能力とコストのバランスが良い）
- チームをできるだけ小さく保つ
- スポーンプロンプトを絞り込んで初期コンテキストを削減
- 作業が完了したらすぐにチームをクリーンアップ

## 6. 制限事項

Agent Teamsは実験的な機能です。現在の既知の制限を理解しておきましょう。

| 制限 | 詳細 |
|------|------|
| **セッション再開不可** | in-processメンバーは`/resume`と`/rewind`で復元されない |
| **タスク状態の遅延** | メンバーがタスクを完了済みにしない場合がある |
| **シャットダウンの遅延** | 現在のリクエストやツール呼び出しを完了してからシャットダウン |
| **1セッション1チーム** | リードは一度に1チームのみ管理可能 |
| **ネストなし** | メンバーはさらにチームやメンバーを生成できない |
| **固定リード** | チームを作成したセッションがそのチームのリードで変更不可 |
| **分割ペインの対応ターミナル** | tmux/iTerm2が必要（VS Code統合ターミナル、Windows Terminal、Ghostty非対応） |

## ハンズオン演習

### 演習 1: Agent Teamsの有効化と基本操作

**目的**: Agent Teamsを有効化し、チームの作成・メンバー生成・タスク割り当てを体験する
**前提条件**: Claude Codeがインストール済みで、中規模以上のプロジェクトで作業していること

**手順**:
1. 環境変数でAgent Teamsを有効化する
   ```bash
   export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
   claude
   ```
2. 簡単なチームを作成する
   ```
   このプロジェクトのコードベースを調査するAgent Teamを作成してください。
   2人のメンバーを生成してください：
   - アーキテクチャ分析を担当する1人
   - テストカバレッジを分析する1人
   ```
3. `Shift+Down`でメンバーを切り替え、それぞれの進捗を確認する
4. 作業完了後、チームをクリーンアップする
   ```
   チームをクリーンアップしてください
   ```

**期待される結果**: 2人のメンバーが独立して調査を行い、リードが結果を統合する。クリーンアップ後にリソースが解放される

### 演習 2: 並列コードレビューの実施

**目的**: 複数の視点からの並列レビューを実践する
**前提条件**: レビュー対象のPRまたはコード変更があること

**手順**:
1. Agent Teamsが有効な状態でClaude Codeを起動する
2. 以下のように並列レビューを依頼する
   ```
   最近のコード変更を3つの観点でレビューするチームを作成してください：
   - セキュリティレビュアー（脆弱性・入力検証に集中）
   - パフォーマンスレビュアー（N+1クエリ・メモリリークに集中）
   - 可読性レビュアー（命名規則・コード構造に集中）
   ```
3. 各レビュアーのフィードバックを確認する
4. リードに結果を統合させる

**期待される結果**: 3つの異なる視点からのレビュー結果が得られ、単一のレビューでは見落としがちな問題が発見される

## よくある質問

**Q: Agent Teamsはどのくらいのコストがかかりますか？**
A: メンバーがプランモードで実行する場合、標準セッションの約7倍のトークンを消費します。3人のメンバーで30分作業すると、通常のセッション数時間分に相当することがあります。必ずコスト意識を持って使用してください。

**Q: in-processモードとsplit-panesモードはどちらがよいですか？**
A: 初めて使う場合はin-processモード（デフォルト）で十分です。複数メンバーの出力を同時に確認したい場合や、直接メンバーを操作したい場合はsplit-panesモードが便利ですが、tmuxまたはiTerm2が必要です。

**Q: メンバーが同じファイルを編集してしまった場合どうなりますか？**
A: 後から書き込んだ方が上書きします。Git上はマージコンフリクトとして検出されますが、Agent Teamsレベルでの自動マージ機能はありません。事前に各メンバーの担当ファイルを明確に分けてください。

**Q: セッションが中断された場合、チームは復元できますか？**
A: 現時点では復元できません。in-processメンバーは`/resume`で復元されない制限があります。重要な作業は途中経過をこまめにコミットするようメンバーに指示してください。

**Q: Agent Teamsとサブエージェントを併用できますか？**
A: はい。各メンバーはサブエージェントを使用できます（ただしメンバーがさらにチームを作ることはできません）。例えば、メンバーがExploreサブエージェントを使ってコードベースを調査することは可能です。

## まとめ

この章で学んだ重要ポイント：

- Agent Teamsは複数のClaude Codeインスタンスが協調する実験的機能（`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`で有効化）
- チームリード・メンバー・タスクリスト・メールボックスの4コンポーネントで構成される
- サブエージェントと異なり、メンバー同士が直接通信し共有タスクリストで自律的に調整できる
- 適切なチームサイズは3〜5人、メンバー当たり5〜6タスクが目安
- トークンコストはシングルセッションの約7倍になり得るため、本当に並列化が必要な場合のみ使用する
