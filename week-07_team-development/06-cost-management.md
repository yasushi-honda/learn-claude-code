# コスト管理・最適化

> 対応する公式ドキュメント: [Manage costs effectively](https://code.claude.com/docs/en/costs)

## 学習目標

- Claude Codeのコスト構造とトークン消費パターンを理解する
- `/cost`コマンドと`/stats`でコストを把握できる
- コスト削減テクニック（MCPサーバー削減・CLAUDE.md最適化・サブエージェント活用）を実践できる
- チームでの支出管理とレート制限の設定方法を習得する
- Agent Teamsのトークンコストを適切に管理できる

## 概要

Claude Codeはインタラクションごとにトークンを消費します。コストはコードベースのサイズ・クエリの複雑さ・会話の長さによって変動します。

**平均的なコスト感:**
- 1日あたりの平均: **約$6/開発者**
- 90%のユーザーの1日コスト: **$12以下**
- Sonnet 4.6使用時の月間平均: **$100〜$200/開発者**（使用頻度・並列インスタンス数・自動化の有無で大きく変動）

## 本文

### コストの確認方法

#### `/cost`コマンド

現在のセッションのトークン使用量統計を表示：

```
/cost
```

出力例：
```
Total cost:            $0.55
Total duration (API):  6m 19.7s
Total duration (wall): 6h 33m 10.2s
Total code changes:    0 lines added, 0 lines removed
```

> **注意**: `/cost`はAPIユーザー向けです。Claude MaxやProサブスクリプションユーザーは`/stats`で使用パターンを確認してください。

#### ステータスラインでの継続監視

コンテキストウィンドウ使用率をリアルタイムで確認するには、カスタムステータスラインを設定します。詳細は[ステータスライン設定](https://code.claude.com/docs/en/statusline)を参照してください。

### チームでのコスト管理

#### ワークスペース支出制限

Claude APIを使用する場合、[コンソール](https://platform.claude.com/docs/en/build-with-claude/workspaces#workspace-limits)でワークスペース全体の支出上限を設定できます。管理者は[使用量とコストのレポート](https://platform.claude.com/docs/en/build-with-claude/workspaces#usage-and-cost-tracking)をコンソールで確認できます。

> **注意**: Claude Codeで認証すると「Claude Code」というワークスペースが自動的に作成されます。このワークスペースはClaude Code認証専用で、APIキーは作成できません。

#### レート制限の推奨値

チームサイズに応じた1ユーザーあたりのToken Per Minute（TPM）とRequest Per Minute（RPM）の推奨値：

| チームサイズ | TPM/ユーザー | RPM/ユーザー |
|------------|------------|------------|
| 1〜5人 | 200k〜300k | 5〜7 |
| 5〜20人 | 100k〜150k | 2.5〜3.5 |
| 20〜50人 | 50k〜75k | 1.25〜1.75 |
| 50〜100人 | 25k〜35k | 0.62〜0.87 |
| 100〜500人 | 15k〜20k | 0.37〜0.47 |
| 500人以上 | 10k〜15k | 0.25〜0.35 |

例: 200ユーザーの場合、ユーザーあたり20k TPM × 200人 = 合計400万TPMを設定。

> **注意**: 大規模なライブトレーニングセッション等の高並列利用が見込まれる場合は、より高いTPMが必要になる可能性があります。

#### Bedrock・Vertex・Foundryでのコスト追跡

これらのプラットフォームではClaudeがクラウドからメトリクスを送信しません。コストメトリクスを取得するには、LiteLLM等のオープンソースツールでキー別の支出を追跡する方法が報告されています（Anthropic非公式・セキュリティ未監査）。

### Agent Teamsのトークンコスト

Agent Teamsは複数のClaude Codeインスタンスを生成し、各インスタンスが独自のコンテキストウィンドウを持ちます。使用量はアクティブなメンバー数に比例します。

メンバーがプランモードで実行する場合、標準セッションの**約7倍**のトークンを消費することがあります。

**コスト管理のポイント:**
- メンバーにはSonnetを使用（能力とコストのバランスが良い）
- チームをできるだけ小さく保つ
- スポーンプロンプトを絞り込んで初期コンテキストを削減
- 作業が完了したらすぐにチームをクリーンアップ

### トークン使用量を削減するテクニック

#### コンテキストをプロアクティブに管理

```bash
# コンテキストウィンドウ使用量を確認
/cost
```

- **タスク間でクリア**: 関連性のない作業に切り替える時は`/clear`でコンテキストをリセット
- **カスタムコンパクション指示**: `/compact コードサンプルとAPI使用法に集中してください`

CLAUDE.mdでコンパクション動作をカスタマイズ：
```markdown
# コンパクション指示

コンパクション時は、テスト出力とコード変更に集中してください
```

#### 適切なモデルを選択

- **ほとんどのコーディングタスク**: Sonnet（コストが低い）
- **複雑なアーキテクチャ決定や多段推論**: Opus
- **単純なサブエージェントタスク**: サブエージェント設定で`model: haiku`を指定

`/model`コマンドでセッション中にモデルを切り替え可能。

#### MCPサーバーのオーバーヘッドを削減

アイドル中でも各MCPサーバーはツール定義をコンテキストに追加します：

```bash
# 現在のコンテキスト使用状況を確認
/context
```

- **MCPより先にCLIツールを優先**: `gh`・`aws`・`gcloud`などはMCPサーバーより軽量
- **未使用サーバーを無効化**: `/mcp`で設定済みサーバーを確認し、使用していないものを無効化
- **ツール検索の自動化**: MCPツールの説明がコンテキストウィンドウの10%を超えると、Claude Codeは自動的にそれらを遅延ロードします

#### コードインテリジェンスプラグインの活用

型付き言語向けのコードインテリジェンスプラグインをインストールすると、Claudeはテキストベースの検索の代わりに精確なシンボルナビゲーションを使用できます。これにより不必要なファイル読み込みが減少します。

#### フックとスキルで処理を委譲

**フックによる前処理:**
大量のログを処理する代わりに、フックでエラー行だけを抽出してClaudeに渡すと、コンテキストを大幅に削減できます。

テスト出力をフィルタリングするフック例：

```json
// settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/filter-test-output.sh"
          }
        ]
      }
    ]
  }
}
```

```bash
#!/bin/bash
# filter-test-output.sh
input=$(cat)
cmd=$(echo "$input" | jq -r '.tool_input.command')

# テスト実行の場合、失敗のみをフィルタリング
if [[ "$cmd" =~ ^(npm test|pytest|go test) ]]; then
  filtered_cmd="$cmd 2>&1 | grep -A 5 -E '(FAIL|ERROR|error:)' | head -100"
  echo "{\"hookSpecificOutput\":{\"hookEventName\":\"PreToolUse\",\"permissionDecision\":\"allow\",\"updatedInput\":{\"command\":\"$filtered_cmd\"}}}"
else
  echo "{}"
fi
```

**スキルによるドメイン知識の提供:**
スキルを使うとオンデマンドでのみ読み込まれるため、常時CLAUDE.mdに含めるより効率的です。

#### CLAUDE.mdをスリム化

CLAUDE.md全体がセッション開始時に読み込まれます。特定のワークフロー向けの詳細な指示はスキルに移動することで、基本コンテキストを小さく保てます。

**目安: CLAUDE.mdは500行以内に**

#### Extended Thinkingの調整

Extended Thinkingはデフォルトで31,999トークンの予算で有効になっています。シンプルなタスクでは：
- `/model`でOpus 4.6のエフォートレベルを下げる
- `/config`でthinkingを無効化
- `MAX_THINKING_TOKENS=8000`でバジェットを削減

#### バーボスな操作をサブエージェントに委譲

テスト実行・ドキュメント取得・ログ処理など大量の出力を生成する操作はサブエージェントに委譲することで、その出力がメインコンテキストに含まれなくなります：

```
テストスイートを実行して失敗したテストとエラーメッセージのみを報告するサブエージェントを使用してください
```

### コスト最適化のためのプロンプト設計

**曖昧なプロンプト（コスト高）:**
```
このコードベースを改善してください
```
→ 広範なスキャンが発生し、多くのトークンを消費

**具体的なプロンプト（コスト低）:**
```
auth.tsのログイン関数に入力バリデーションを追加してください
```
→ Claudeが効率的に最小限のファイル読み込みで作業できる

### バックグラウンドのトークン消費

アイドル中でもClaude Codeは以下でトークンを使用します：
- **会話のサマリー化**: `claude --resume`機能のためのバックグラウンドジョブ
- **コマンド処理**: `/cost`等のコマンドがステータス確認リクエストを生成することがある

これらのバックグラウンド処理は通常セッションあたり$0.04未満です。

### コスト管理チェックリスト

#### 個人での利用

- [ ] ステータスラインでコンテキスト使用率を常時監視
- [ ] 無関係なタスクに切り替える前に`/clear`を実行
- [ ] 未使用のMCPサーバーを`/mcp`で無効化
- [ ] ほとんどのタスクにSonnetを使用し、Opusは複雑な推論にのみ使用
- [ ] 具体的なプロンプトを心がける

#### チームでの利用

- [ ] コンソールでワークスペース支出上限を設定
- [ ] チームサイズに応じたTPM/RPM上限を設定
- [ ] 組織全体のCLAUDE.mdを`~/.claude/`に配置（共通の指示）
- [ ] バッチ処理には`--max-budget-usd`フラグを設定
- [ ] Agent Teamsのコストを定期的に確認

## まとめ

- Claude Codeの平均コストは1日$6/開発者、月間$100〜$200/開発者（使用状況による）
- `/cost`コマンドで現在のセッションのトークン使用量を確認できる
- コスト削減の主なアプローチ: コンテキスト管理・適切なモデル選択・MCPサーバー削減・具体的なプロンプト
- Agent Teamsはシングルセッションの約7倍のトークンを消費するため、本当に必要な場面でのみ使用する
- チーム管理者はコンソールで支出上限とTPM/RPM上限を設定できる
- フックとスキルを活用してClaudeが処理する前にデータを前処理し、コンテキストを削減できる
- CLAUDE.mdは500行以内に保ち、詳細な指示はスキルに移動する
