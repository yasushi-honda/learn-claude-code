## この章の学習目標

この章を終えると、以下のことができるようになります：

1. Claude Codeのコスト構造（平均$6/日/開発者、月間$100〜$200）を理解し、見積もりができる
2. `/cost`コマンドと`/stats`でセッションのコストをリアルタイムに把握できる
3. コンテキスト管理・モデル選択・MCPサーバー削減など、具体的なコスト削減テクニックを実践できる
4. チームでのレート制限設定と支出管理の方法を習得できる

## 1. コスト構造の理解

### 平均的なコスト

Claude Codeはインタラクションごとにトークンを消費します。公式が示す典型的なコスト感は以下の通りです。

| 指標 | コスト |
|------|--------|
| 1日あたりの平均 | **約$6/開発者** |
| 90%のユーザーの1日コスト | **$12以下** |
| Sonnet 4.6使用時の月間平均 | **$100〜$200/開発者** |

月間コストは使用頻度・並列インスタンス数・自動化の有無で大きく変動します。

### モデル別の単価

| モデル | 用途 | Input / Output per MTok | Cache Write / Read |
|--------|------|------------------------|-------------------|
| **Opus 4.6** | 深い推論、設計判断 | $15 / $75 | $18.75 / $1.875 |
| **Sonnet 4.6** | コーディング全般 | $3 / $15 | $3.75 / $0.375 |
| **Haiku 4.5** | 機械的タスクのみ | $1 / $5 | $1.25 / $0.10 |

Sonnet 4.6はSWE-benchでOpus 4.6と1.2pt差（79.6% vs 80.8%）でありながら、コストは約1/5です。ほとんどのコーディングタスクにはSonnetが推奨されます。

### Agent Teamsのトークンコスト

Agent Teamsは複数のClaude Codeインスタンスを生成し、各インスタンスが独自のコンテキストウィンドウを持ちます。メンバーがプランモードで実行する場合、標準セッションの**約7倍**のトークンを消費することがあります。

### バックグラウンドのトークン消費

アイドル中でもClaude Codeは以下でトークンを使用します：
- 会話のサマリー化（`claude --resume`機能のため）
- コマンド処理のステータス確認

これらは通常セッションあたり**$0.04未満**です。

## 2. コストの確認方法

### `/cost`コマンド

現在のセッションのトークン使用量統計を表示します。

```
/cost
```

出力例：
```
Total cost:            $0.55
Total duration (API):  6m 19.7s
Total duration (wall): 6h 33m 10.2s
Total code changes:    0 lines added, 0 lines removed
```

> **公式ドキュメントより**: `/cost`はAPIユーザー向けです。Claude MaxやProサブスクリプションユーザーは`/stats`で使用パターンを確認してください。

### ステータスラインでの継続監視

コンテキストウィンドウ使用率をリアルタイムで確認するには、カスタムステータスラインを設定します。使用率が70〜80%を超えると品質劣化が始まるため、早めの対応が重要です。

### ワークスペース支出制限

Claude APIを使用する場合、[Anthropicコンソール](https://platform.claude.com)でワークスペース全体の支出上限を設定できます。管理者はコンソールで使用量とコストのレポートを確認できます。

## 3. チームでのレート制限

チームサイズに応じた1ユーザーあたりのToken Per Minute（TPM）とRequest Per Minute（RPM）の推奨値です。

| チームサイズ | TPM/ユーザー | RPM/ユーザー |
|------------|------------|------------|
| 1〜5人 | 200k〜300k | 5〜7 |
| 5〜20人 | 100k〜150k | 2.5〜3.5 |
| 20〜50人 | 50k〜75k | 1.25〜1.75 |
| 50〜100人 | 25k〜35k | 0.62〜0.87 |
| 100〜500人 | 15k〜20k | 0.37〜0.47 |
| 500人以上 | 10k〜15k | 0.25〜0.35 |

**計算例**: 200ユーザーの場合、ユーザーあたり20k TPM x 200人 = 合計400万TPMを設定。

> **公式ドキュメントより**: 大規模なライブトレーニングセッション等の高並列利用が見込まれる場合は、より高いTPMが必要になる可能性があります。

### Bedrock・Vertex・Foundryでのコスト追跡

これらのプラットフォームではClaudeがクラウドからメトリクスを送信しません。LiteLLM等のオープンソースツールでキー別の支出を追跡する方法が報告されています（Anthropic非公式・セキュリティ未監査）。

## 4. コスト削減テクニック

### テクニック1: コンテキストをプロアクティブに管理

コンテキストウィンドウの使用量がコストに直結します。

**タスク間でクリア:**
```
/clear
```
関連性のない作業に切り替える時は`/clear`でコンテキストをリセットしてください。

**カスタムコンパクション指示:**
```
/compact コードサンプルとAPI使用法に集中してください
```

CLAUDE.mdでコンパクション動作をカスタマイズすることもできます：
```markdown
# コンパクション指示

コンパクション時は、テスト出力とコード変更に集中してください
```

### テクニック2: 適切なモデルを選択

| 用途 | 推奨モデル | 理由 |
|------|----------|------|
| ほとんどのコーディングタスク | Sonnet | コスト1/5でほぼ同等の性能 |
| 複雑なアーキテクチャ決定 | Opus | 深い推論が必要 |
| 単純なサブエージェントタスク | Haiku | `model: haiku`で指定 |

`/model`コマンドでセッション中にモデルを切り替え可能です。

### テクニック3: MCPサーバーのオーバーヘッドを削減

アイドル中でも各MCPサーバーはツール定義をコンテキストに追加します。

```
/context
```

で現在のコンテキスト使用状況を確認してください。

**実践的な対策：**
- **CLIツールを優先**: `gh`、`aws`、`gcloud`等のCLIツールはMCPサーバーより軽量
- **未使用サーバーを無効化**: `/mcp`で設定済みサーバーを確認し、使用していないものを無効化
- **自動遅延ロード**: MCPツールの説明がコンテキストウィンドウの10%を超えると、Claude Codeは自動的にそれらを遅延ロードする

### テクニック4: コードインテリジェンスプラグインの活用

型付き言語向けのコードインテリジェンスプラグインをインストールすると、Claudeはテキストベースの検索の代わりに精確なシンボルナビゲーションを使用できます。不必要なファイル読み込みが減少し、トークンが節約されます。

### テクニック5: フックとスキルで処理を委譲

**フックによる前処理：**

大量のログを処理する代わりに、フックでエラー行だけを抽出してClaudeに渡すと、コンテキストを大幅に削減できます。

テスト出力をフィルタリングするフック例：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/filter-test-output.sh"
          }
        ]
      }
    ]
  }
}
```

```bash
#!/bin/bash
# filter-test-output.sh
input=$(cat)
cmd=$(echo "$input" | jq -r '.tool_input.command')

# テスト実行の場合、失敗のみをフィルタリング
if [[ "$cmd" =~ ^(npm test|pytest|go test) ]]; then
  filtered_cmd="$cmd 2>&1 | grep -A 5 -E '(FAIL|ERROR|error:)' | head -100"
  echo "{\"hookSpecificOutput\":{\"hookEventName\":\"PreToolUse\",\"permissionDecision\":\"allow\",\"updatedInput\":{\"command\":\"$filtered_cmd\"}}}"
else
  echo "{}"
fi
```

**スキルによるドメイン知識の提供：**

スキルを使うとオンデマンドでのみ読み込まれるため、常時CLAUDE.mdに含めるより効率的です。

### テクニック6: CLAUDE.mdをスリム化

CLAUDE.md全体がセッション開始時に読み込まれます。特定のワークフロー向けの詳細な指示はスキルに移動することで、基本コンテキストを小さく保てます。

**目安: CLAUDE.mdは500行以内に**

### テクニック7: Extended Thinkingの調整

Extended Thinkingはデフォルトで31,999トークンの予算で有効になっています。シンプルなタスクでは以下の調整が有効です：

- `/model`でOpus 4.6のエフォートレベルを下げる
- `/config`でthinkingを無効化
- `MAX_THINKING_TOKENS=8000`でバジェットを削減

### テクニック8: バーボスな操作をサブエージェントに委譲

テスト実行・ドキュメント取得・ログ処理など大量の出力を生成する操作はサブエージェントに委譲することで、その出力がメインコンテキストに含まれなくなります。

```
テストスイートを実行して失敗したテストとエラーメッセージのみを報告するサブエージェントを使用してください
```

### テクニック9: 具体的なプロンプトを心がける

**曖昧なプロンプト（コスト高）:**
```
このコードベースを改善してください
```
→ 広範なスキャンが発生し、多くのトークンを消費

**具体的なプロンプト（コスト低）:**
```
auth.tsのログイン関数に入力バリデーションを追加してください
```
→ Claudeが効率的に最小限のファイル読み込みで作業できる

## ハンズオン演習

### 演習 1: セッションコストの計測

**目的**: `/cost`コマンドでコストを計測し、異なる操作のコスト感を体感する
**前提条件**: Claude Codeがインストール済みで、APIキーが設定されていること

**手順**:
1. Claude Codeを起動して`/cost`を確認する（初期状態）
2. 簡単な質問をする
   ```
   このプロジェクトの主なファイル構成を教えてください
   ```
3. `/cost`を再度確認し、トークン消費量を記録する
4. コードベースの広範な調査を依頼する
   ```
   このプロジェクトの全ファイルの依存関係グラフを作成してください
   ```
5. `/cost`を再度確認し、広範な調査のコスト増加を比較する
6. `/clear`でコンテキストをリセットし、`/cost`を確認する

**期待される結果**: 簡単な質問と広範な調査でコストが大きく異なることを体感できる。`/clear`後にコンテキスト使用量がリセットされることを確認できる

### 演習 2: モデル切り替えによるコスト最適化

**目的**: モデル選択がコストに与える影響を比較する

**手順**:
1. Opusモデルで質問する
   ```
   /model opus
   ```
   ```
   auth.tsの関数を一覧してください
   ```
2. `/cost`でコストを確認する
3. Sonnetモデルに切り替えて同じ質問をする
   ```
   /model sonnet
   ```
   ```
   auth.tsの関数を一覧してください
   ```
4. `/cost`でコストを比較する

**期待される結果**: 同じタスクでもOpusとSonnetでコストが大幅に異なることを確認できる

### 演習 3: テスト出力フィルタリングフックの作成

**目的**: フックでテスト出力をフィルタリングし、コンテキスト消費を削減する

**手順**:
1. `~/.claude/hooks/filter-test-output.sh`を作成する（本章のフック例を使用）
2. `chmod +x`で実行権限を付与する
3. `settings.json`にフック設定を追加する
4. テストを実行して、フィルタリングされた出力のみがコンテキストに含まれることを確認する
5. `/cost`でフィルタリングなしの場合と比較する

**期待される結果**: テスト出力が失敗箇所のみにフィルタリングされ、コンテキスト消費が削減される

## よくある質問

**Q: Claude MaxやProサブスクリプションの場合、コストを気にする必要はありますか？**
A: サブスクリプションプランではAPI課金は発生しませんが、レート制限があります。コンテキスト管理のベストプラクティス（`/clear`、具体的なプロンプト等）は応答品質の維持に重要です。`/stats`で使用パターンを確認してください。

**Q: Agent Teamsのコストをどう見積もればよいですか？**
A: メンバー数 x 標準セッションコストの約7倍（プランモード時）が目安です。3人のメンバーで30分作業すると、$6〜$12程度になることがあります。本当に並列化が必要か、サブエージェントで代替できないかを先に検討してください。

**Q: コスト上限を超えないようにする方法はありますか？**
A: `--max-budget-usd`フラグ（headless実行時）、ワークスペース支出上限（コンソール設定）、TPM/RPMレート制限の3つの方法があります。CI/CDでは`--max-budget-usd`と`--max-turns`の併用を推奨します。

**Q: MCPサーバーのコンテキスト消費量はどのくらいですか？**
A: サーバーごとに異なりますが、ツール定義が多いMCPサーバーは数千トークンをコンテキストに追加します。`/context`で確認し、コンテキストの10%以上を占める場合は自動遅延ロードが有効になります。

**Q: CLAUDE.mdを500行以内に保つコツはありますか？**
A: 頻繁に使う基本ルールのみCLAUDE.mdに残し、特定のワークフロー向けの詳細な指示（コーディング規約の詳細、テスト戦略など）はスキルに移動してください。スキルはオンデマンドでのみ読み込まれるため、基本コンテキストを圧迫しません。

## まとめ

この章で学んだ重要ポイント：

- Claude Codeの平均コストは1日$6/開発者、月間$100〜$200/開発者（Sonnet 4.6使用時）
- `/cost`でセッションのトークン使用量を確認、サブスクリプションユーザーは`/stats`を使用
- コスト削減の主なアプローチ: `/clear`によるコンテキスト管理、Sonnetの積極使用、MCPサーバー削減、具体的なプロンプト
- Agent Teamsはシングルセッションの約7倍のトークンを消費するため、本当に必要な場面でのみ使用
- チーム管理者はコンソールで支出上限、TPM/RPM制限を設定できる
- フック・スキル・サブエージェントを活用してClaudeが処理するデータ量を削減する
