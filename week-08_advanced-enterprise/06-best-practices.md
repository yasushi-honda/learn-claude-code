# ベストプラクティス総まとめ

> 対応する公式ドキュメント: [Best Practices for Claude Code](https://code.claude.com/docs/en/best-practices)

## 学習目標

- Claude Codeのコンテキストウィンドウ管理の重要性を理解する
- 検証可能なタスク設計によってClaude Codeの出力品質を高められる
- 探索→計画→実装→コミットの4フェーズワークフローを実践できる
- CLAUDE.mdを効果的に設計・維持できる
- 並列セッションとファンアウトパターンで作業をスケールさせる方法を把握する

## 概要

Claude Codeのベストプラクティスは、ほぼすべてひとつの制約に基づいています：**コンテキストウィンドウはすぐに埋まり、埋まるほど性能が劣化する**という事実です。Anthropicの内部チームや様々なコードベース・言語・環境でClaude Codeを使うエンジニアが実証した効果的なパターンをまとめます。

## 本文

### コンテキストウィンドウ管理の重要性

コンテキストウィンドウはClaude Codeで最も重要なリソースです：

- 会話全体（メッセージ・ファイル内容・コマンド出力）がコンテキストを消費
- 1回のデバッグセッションや探索で数万トークンを消費することも
- **LLMはコンテキストが埋まるほど性能が劣化**し、過去の指示を「忘れる」ことがある

**対策:**

```bash
# コンテキスト使用率を常時監視（カスタムstatus line）
/compact          # 手動でコンテキストを圧縮
/clear            # 無関係なタスク間でコンテキストをリセット
/compact 特定の指示  # フォーカスを絞って圧縮
```

---

### 検証可能なタスク設計

**Claude Codeが自分の作業を確認できるようにすることが、単一の最高効果のある方法です。**

テスト・スクリーンショット・期待出力を提供することで、Claudeは自己確認できます：

| 戦略 | NG例 | OK例 |
|-----|-----|-----|
| **検証基準を提供** | "メールアドレス検証関数を実装して" | "validateEmail関数を実装して。テストケース: user@example.comはtrue、invalidはfalse。実装後にテストを実行して" |
| **UI変更を視覚的に確認** | "ダッシュボードを改善して" | "[スクリーンショット貼付] このデザインを実装して。結果のスクリーンショットを撮って比較し、差異を修正して" |
| **症状ではなく根本原因に対処** | "ビルドが失敗している" | "ビルドがこのエラーで失敗しています：[エラー貼付]。修正してビルド成功を確認して。エラーを抑制するのではなく根本原因に対処して" |

UIの変更確認には[Claude in Chrome拡張機能](https://code.claude.com/docs/en/chrome)が便利です。

### 探索→計画→実装→コミットの4フェーズワークフロー

すぐコードを書かせると「間違った問題を解決する」リスクがあります。[Plan Mode](https://code.claude.com/docs/en/common-workflows#use-plan-mode-for-safe-code-analysis)で探索と実行を分離します：

**フェーズ1: 探索（Plan Modeで）**

```text
/src/authを読んで、セッションとログイン処理の仕組みを理解して。
環境変数でのシークレット管理方法も確認して。
```

**フェーズ2: 計画（Plan Modeで）**

```text
Google OAuthを追加したい。どのファイルを変更する必要がある？
セッションフローは？計画を作って。
```

`Ctrl+G`でプランをテキストエディタで直接編集できます。

**フェーズ3: 実装（通常モードで）**

```text
プランに従ってOAuthフローを実装して。コールバックハンドラーのテストを書いて、
テストスイートを実行して失敗を修正して。
```

**フェーズ4: コミット**

```text
説明的なメッセージでコミットしてPRを作成して
```

> **注意**: Plan Modeはオーバーヘッドを追加します。タイポ修正・ログ行追加・変数リネームなど、スコープが明確で小さな修正はPlanModeなしで直接依頼してください。

### 効果的なプロンプト設計

**指示が具体的なほど、修正回数が減ります。**

| 戦略 | NG例 | OK例 |
|-----|-----|-----|
| **タスクをスコープ** | "foo.pyのテストを追加して" | "foo.pyのユーザーがログアウトしているエッジケースのテストを書いて。モックは使わないで" |
| **ソースを指定** | "なぜExecutionFactoryのAPIはこんなに変なの？" | "ExecutionFactoryのgit履歴を調べて、APIがどうやってこうなったかまとめて" |
| **既存パターンを参照** | "カレンダーウィジェットを追加して" | "HotDogWidget.phpのパターンを参考に、同じパターンでユーザーが月を選択し年を前後にページネーションできるカレンダーウィジェットを実装して" |

**リッチなコンテキスト提供方法:**

```bash
# @でファイルを参照
@src/auth/login.ts を見てこの関数を理解して

# 画像を直接ペースト（スクリーンショット、ワイヤーフレーム等）

# URLを提供（ドキュメント、API参照）

# データをパイプ
cat error.log | claude

# Claude自身に必要なコンテキストを取得させる
Bash コマンドやMCPツールを使って必要な情報を自分で取得して
```

### 効果的なCLAUDE.mdの設計

**`/init`を実行してCLAUDE.mdのスターターを生成し、その後継続的に改善します。**

#### 含めるべき内容・含めるべきでない内容

| 含めるべき内容 | 含めるべきでない内容 |
|-------------|-----------------|
| Claudeが推測できないBashコマンド | コードを読めばわかること |
| デフォルトと異なるコードスタイルルール | Claude が既に知っている標準的な言語規約 |
| テスト指示と優先テストランナー | 詳細なAPIドキュメント（代わりにリンクを貼る） |
| リポジトリのエチケット（ブランチ命名・PR規約） | 頻繁に変わる情報 |
| プロジェクト固有のアーキテクチャの決定 | 長い説明やチュートリアル |
| 開発環境の注意点（必要な環境変数） | コードベースのファイルごとの説明 |
| 見落としやすい非自明な動作 | 「クリーンなコードを書く」などの自明なプラクティス |

**CLAUDE.md例:**

```markdown
# コードスタイル
- ESモジュール（import/export）構文を使用、CommonJS（require）は不可
- 可能な場合はインポートを分割代入する（例: import { foo } from 'bar'）

# ワークフロー
- 一連のコード変更が完了したら必ずタイプチェックを実行
- パフォーマンスのため、テストスイート全体ではなく単一テストを優先して実行
```

#### CLAUDE.mdのインポート構文

```markdown
CLAUDE.md

プロジェクト概要は @README.md を、利用可能なnpmコマンドは @package.json を参照。

# 追加指示
- Gitワークフロー: @docs/git-instructions.md
- 個人オーバーライド: @~/.claude/my-project-instructions.md
```

#### 配置場所

```
~/.claude/CLAUDE.md        ← 全Claudeセッションに適用（個人設定）
./CLAUDE.md                ← プロジェクトルート（チームで共有、git管理推奨）
./CLAUDE.local.md          ← プロジェクトルート（個人設定、.gitignore対象）
parent/CLAUDE.md           ← モノレポでの親ディレクトリ設定
child-dir/CLAUDE.md        ← サブディレクトリの作業時に自動読み込み
```

#### CLAUDE.mdの維持方法

- **肥大化サイン**: Claudeが指示を無視し始めたらファイルが長すぎる可能性
- **定期的な刈り込み**: 各行に「これがなければClaudeはミスをするか？」と自問
- **フックへの移行**: Claude がすでに正しく行っていることはルールから削除し、確実に実行させたい場合はフックに変換
- **emphasisの活用**: "IMPORTANT"や"YOU MUST"で重要なルールを強調

### パーミッション設定

```json
// settings.json - 安全なコマンドの許可リスト
{
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(git diff *)",
      "Bash(git status)",
      "Bash(npm test)"
    ]
  }
}
```

または`/sandbox`でOSレベルの分離を有効化。`--dangerously-skip-permissions`はコンテナなどの安全な環境でのみ使用してください。

### セッション管理

#### 早期の軌道修正

- **`Esc`**: Claudeを途中で停止。コンテキストは保持されるため、方向転換が可能
- **`Esc + Esc`または`/rewind`**: 以前の会話とコード状態を復元
- **`"Undo that"`**: Claudeに変更を元に戻させる
- **`/clear`**: 無関係なタスク間でコンテキストをリセット

**同じ問題で2回修正しても改善しない場合**: `/clear`してより具体的な初期プロンプトで新規セッションを開始する。

#### コンテキストの積極的な管理

- 無関係なタスク間では`/clear`を使用
- 自動コンパクションでは重要なコードと決定が保持される
- `"/compact API変更に集中して"`のように焦点を絞ったコンパクション
- `Esc + Esc`の後に**Summarize from here**で会話の一部のみ要約
- CLAUDE.mdに圧縮の指示を含める：`"圧縮時は常に変更ファイルの一覧とテストコマンドを保持"`

#### セッションの再開

```bash
claude --continue    # 最後の会話を再開
claude --resume      # 最近の会話から選択して再開
/rename              # セッションに"oauth-migration"等の名前をつける
```

### サブエージェントの活用

コンテキスト消費を抑えながら調査作業を実行するのにサブエージェントは最強のツールです：

```text
# サブエージェントで調査（メインコンテキストを汚染しない）
サブエージェントを使って認証システムのトークンリフレッシュ処理と、
再利用できる既存のOAuthユーティリティがあるかを調査して。

# 実装後の検証
サブエージェントを使ってこのコードのエッジケースをレビューして
```

---

### チェックポイントとリワインド

Claude は変更前に自動的にチェックポイントを作成します：

- `Esc + Esc`または`/rewind`でリワインドメニューを開く
- 会話のみ・コードのみ・両方の復元が可能
- チェックポイントはセッションをまたいで保持される

> **注意**: チェックポイントはClaude が行った変更のみを追跡します。外部プロセスによる変更は含まれません。gitの代替ではありません。

### 自動化とスケールアップ

#### 非インタラクティブモード

```bash
# 単発クエリ
claude -p "このプロジェクトが何をするか説明して"

# スクリプト向け構造化出力
claude -p "全APIエンドポイントを一覧表示して" --output-format json

# リアルタイム処理向けストリーミング
claude -p "このログファイルを分析して" --output-format stream-json
```

#### 並列セッションの活用パターン

**Writer/Reviewerパターン:**

| セッションA（Writer） | セッションB（Reviewer） |
|---------------------|---------------------|
| `APIエンドポイントにレート制限を実装して` | |
| | `@src/middleware/rateLimiter.ts のレート制限実装をレビューして。エッジケース・レース条件・既存middlewareとの整合性を確認して` |
| `レビューフィードバック：[セッションBの出力]。これらの問題に対処して` | |

**ファンアウトパターン（大規模マイグレーション）:**

```bash
# 1. タスクリストを生成
# Claudeに "マイグレーションが必要な全Pythonファイルを一覧表示して" を依頼

# 2. ループスクリプトを作成
for file in $(cat files.txt); do
  claude -p "ReactからVueへ${file}をマイグレートして。OKまたはFAILを返して" \
    --allowedTools "Edit,Bash(git commit *)"
done

# 3. 最初の2〜3ファイルで検証してから全体に展開
```

### よくある失敗パターンと対策

| 失敗パターン | 対策 |
|------------|------|
| **キッチンシンクセッション**: 無関係なタスクが混在してコンテキストが雑然 | 無関係なタスク間で`/clear` |
| **繰り返し修正**: 同じ問題を何度も修正するとコンテキストが失敗例で汚染 | 2回修正しても改善しないなら`/clear`してより具体的な初期プロンプトで再開 |
| **肥大化したCLAUDE.md**: 長すぎて重要なルールが埋もれる | 容赦なく刈り込む。Claudeが既にできることは削除するかフックに変換 |
| **確認なしの信頼**: 正しく見えるが機能しない実装 | 常に検証（テスト・スクリプト・スクリーンショット）を提供 |
| **無限探索**: スコープなしで「調査して」と頼むとファイルを大量に読む | 調査の範囲を狭く指定するか、サブエージェントを使う |

---

### 直感の発展

これらのパターンは出発点であり、すべての状況に最適ではありません。状況によっては：

- **コンテキストを積み上げるべき場合**: 複雑な一つの問題を深く掘り下げているとき
- **計画なしで始めるべき場合**: 探索的なタスクでClaudeの解釈を見たいとき
- **漠然としたプロンプトが適切な場合**: Claudeに問題を自由に解釈させたいとき

良い出力が得られたとき何をしたかを記録し、Claude が苦労するとき何が原因かを分析することで、独自のベストプラクティスが構築されます。

## まとめ

- コンテキストウィンドウが最重要リソース。`/clear`・`/compact`・サブエージェントで積極管理する
- 検証可能なタスク設計（テスト・スクリーンショット・期待出力）がClaude Codeの品質を最大化する
- Plan Modeで探索→計画→実装→コミットの4フェーズを守り、「間違った問題の解決」を防ぐ
- CLAUDE.mdは短く・具体的に。不要な行は削除し、確実性が必要なものはフックに移行する
- サブエージェントを使ってメインコンテキストを汚染せずに調査や検証を実行する
- `claude -p`とファンアウトパターンで大規模マイグレーションや分析をスケールできる
- 同じ問題で2回失敗したら`/clear`して新鮮な視点でアプローチし直す
