> **対応公式ドキュメント**: https://code.claude.com/docs/en/best-practices
> **想定所要時間**: 約60分
> **難易度**: ★★★☆☆

## この章の学習目標

この章を終えると、以下のことができるようになります：

1. コンテキストウィンドウの管理が品質に直結することを理解し、適切に管理できる
2. 検証可能なタスク設計によってClaude Codeの出力品質を最大化できる
3. 探索→計画→実装→コミットの4フェーズワークフローを実践できる
4. CLAUDE.mdの効果的な設計・維持・配置ができる
5. 並列セッションとファンアウトパターンで作業をスケールさせる方法を把握している

## 1. コンテキストウィンドウ管理

> **公式ドキュメントより**: Claude Codeのベストプラクティスは、ほぼすべてひとつの制約に基づいています：コンテキストウィンドウはすぐに埋まり、埋まるほど性能が劣化するという事実です。

### コンテキストウィンドウとは

コンテキストウィンドウには、会話全体（メッセージ・ファイル内容・コマンド出力）が含まれます：

- 1回のデバッグセッションや探索で数万トークンを消費する場合がある
- **LLMはコンテキストが埋まるほど性能が劣化**し、過去の指示を「忘れる」ことがある
- Claude Codeは使用率が70-80%で品質劣化が始まる

### コンテキスト管理のコマンド

```bash
/compact          # 手動でコンテキストを圧縮
/compact 特定の指示  # フォーカスを絞って圧縮（例: "/compact API変更に集中して"）
/clear            # 無関係なタスク間でコンテキストをリセット
/rewind           # 以前の会話とコード状態を復元
/rename           # セッションに名前をつける（例: "oauth-migration"）
```

### セッション管理のベストプラクティス

| 状況 | 対応 |
|------|------|
| 新しいタスクに移る | `/clear` でコンテキストをリセット |
| コンテキスト使用率70%超 | `/compact` で圧縮 |
| 同じ問題で2回修正しても改善しない | `/clear` してより具体的な初期プロンプトで再開 |
| セッションを後で再開したい | `claude --continue` または `claude --resume` |

## 2. 検証可能なタスク設計

**Claude Codeが自分の作業を確認できるようにすることが、単一の最高効果のある方法です。**

### 検証手段を与える

テスト・スクリーンショット・期待出力を提供することで、Claudeは自己確認できます：

| 戦略 | NG例 | OK例 |
|-----|-----|-----|
| **検証基準を提供** | "メールアドレス検証関数を実装して" | "validateEmail関数を実装して。テストケース: user@example.comはtrue、invalidはfalse。実装後にテストを実行して" |
| **UI変更を視覚的に確認** | "ダッシュボードを改善して" | "[スクリーンショット貼付] このデザインを実装して。結果のスクリーンショットを撮って比較して" |
| **症状ではなく根本原因に対処** | "ビルドが失敗している" | "ビルドがこのエラーで失敗しています：[エラー貼付]。根本原因に対処して、エラーを抑制しないで" |

## 3. 探索→計画→実装→コミットの4フェーズ

すぐコードを書かせると「間違った問題を解決する」リスクがあります。[Plan Mode](https://code.claude.com/docs/en/common-workflows#use-plan-mode-for-safe-code-analysis) で探索と実行を分離します。

### フェーズ1: 探索（Plan Modeで）

```text
/src/authを読んで、セッションとログイン処理の仕組みを理解して。
環境変数でのシークレット管理方法も確認して。
```

### フェーズ2: 計画（Plan Modeで）

```text
Google OAuthを追加したい。どのファイルを変更する必要がある？
セッションフローは？計画を作って。
```

`Ctrl+G` でプランをテキストエディタで直接編集できます。

### フェーズ3: 実装（通常モードで）

```text
プランに従ってOAuthフローを実装して。コールバックハンドラーのテストを書いて、
テストスイートを実行して失敗を修正して。
```

### フェーズ4: コミット

```text
説明的なメッセージでコミットしてPRを作成して
```

> **注意**: Plan Modeはオーバーヘッドを追加します。タイポ修正・ログ行追加・変数リネームなど、スコープが明確で小さな修正はPlan Modeなしで直接依頼してください。

## 4. 効果的なプロンプト設計

### 具体的なコンテキストを提供する

指示が具体的なほど、修正回数が減ります：

| 戦略 | NG例 | OK例 |
|-----|-----|-----|
| **タスクをスコープ** | "foo.pyのテストを追加して" | "foo.pyのユーザーがログアウトしているエッジケースのテストを書いて。モックは使わないで" |
| **ソースを指定** | "なぜこのAPIは変なの？" | "ExecutionFactoryのgit履歴を調べて、APIがどうなったかまとめて" |
| **既存パターンを参照** | "カレンダーウィジェットを追加して" | "HotDogWidget.phpのパターンを参考に、同じパターンでカレンダーウィジェットを実装して" |

### リッチなコンテキスト提供方法

```bash
# @でファイルを参照
@src/auth/login.ts を見てこの関数を理解して

# 画像を直接ペースト（スクリーンショット、ワイヤーフレーム等）

# URLを提供（ドキュメント、API参照）

# データをパイプ
cat error.log | claude

# Claude自身に必要なコンテキストを取得させる
Bashコマンドを使って必要な情報を自分で取得して
```

### インタビュー方式

情報が不足している場合、Claudeに質問させることもできます：

```text
AskUserQuestionツールを使って、要件を詳細にインタビューして
```

## 5. 効果的なCLAUDE.mdの設計

### `/init` でスターターを生成

`/init` コマンドを実行すると、プロジェクトに合わせたCLAUDE.mdのスターターが生成されます。その後、継続的に改善していきます。

### 含めるべき内容・含めるべきでない内容

| 含めるべき内容 | 含めるべきでない内容 |
|-------------|-----------------|
| Claudeが推測できないBashコマンド | コードを読めばわかること |
| デフォルトと異なるコードスタイルルール | 標準的な言語規約 |
| テスト指示と優先テストランナー | 詳細なAPIドキュメント（リンクで代替） |
| リポジトリのエチケット（ブランチ命名・PR規約） | 頻繁に変わる情報 |
| プロジェクト固有のアーキテクチャ決定 | 長い説明やチュートリアル |
| 開発環境の注意点（必要な環境変数） | コードベースのファイルごとの説明 |
| 見落としやすい非自明な動作 | 「クリーンなコードを書く」などの自明なプラクティス |

### CLAUDE.mdの例

```markdown
# コードスタイル
- ESモジュール（import/export）構文を使用、CommonJS（require）は不可
- 可能な場合はインポートを分割代入する

# ワークフロー
- 一連のコード変更が完了したら必ずタイプチェックを実行
- パフォーマンスのため、テストスイート全体ではなく単一テストを優先して実行
```

### @import構文

```markdown
プロジェクト概要は @README.md を、npmコマンドは @package.json を参照。

# 追加指示
- Gitワークフロー: @docs/git-instructions.md
- 個人オーバーライド: @~/.claude/my-project-instructions.md
```

### 配置場所

```
~/.claude/CLAUDE.md        <-- 全Claudeセッションに適用（個人設定）
./CLAUDE.md                <-- プロジェクトルート（チームで共有、git管理推奨）
./CLAUDE.local.md          <-- プロジェクトルート（個人設定、.gitignore対象）
parent/CLAUDE.md           <-- モノレポでの親ディレクトリ設定
child-dir/CLAUDE.md        <-- サブディレクトリの作業時に自動読み込み
```

### 維持のポイント

- **肥大化サイン**: Claudeが指示を無視し始めたらファイルが長すぎる可能性
- **定期的な刈り込み**: 各行に「これがなければClaudeはミスをするか？」と自問
- **フックへの移行**: Claudeが既に正しく行っていることは削除し、確実に実行させたい場合はフックに変換
- **emphasisの活用**: "IMPORTANT" や "YOU MUST" で重要なルールを強調

## 6. サブエージェントとCLIツールの活用

### サブエージェント

コンテキスト消費を抑えながら調査作業を実行するのにサブエージェントは最強のツールです：

```text
# サブエージェントで調査（メインコンテキストを汚染しない）
サブエージェントを使って認証システムのトークンリフレッシュ処理と、
再利用できる既存のOAuthユーティリティがあるかを調査して。

# 実装後の検証
サブエージェントを使ってこのコードのエッジケースをレビューして
```

### CLIツール活用

MCPサーバーより軽量なCLIツールを優先します：

```bash
# GitHub操作
gh pr create --title "Add OAuth" --body "..."
gh issue list --assignee @me

# AWS操作
aws bedrock list-inference-profiles --region us-east-1

# GCP操作
gcloud services enable aiplatform.googleapis.com
```

## 7. 自動化とスケールアップ

### 非インタラクティブモード

```bash
# 単発クエリ
claude -p "このプロジェクトが何をするか説明して"

# スクリプト向け構造化出力
claude -p "全APIエンドポイントを一覧表示して" --output-format json

# リアルタイム処理向けストリーミング
claude -p "このログファイルを分析して" --output-format stream-json
```

### ファンアウトパターン（大規模マイグレーション）

```bash
# 1. タスクリストを生成
# Claudeに "マイグレーションが必要な全Pythonファイルを一覧表示して" を依頼

# 2. ループスクリプトを作成
for file in $(cat files.txt); do
  claude -p "ReactからVueへ${file}をマイグレートして。OKまたはFAILを返して" \
    --allowedTools "Edit,Bash(git commit *)"
done

# 3. 最初の2-3ファイルで検証してから全体に展開
```

### Writer/Reviewerパターン

| セッションA（Writer） | セッションB（Reviewer） |
|---------------------|---------------------|
| `APIエンドポイントにレート制限を実装して` | |
| | `@src/middleware/rateLimiter.ts のレート制限実装をレビューして` |
| `レビューフィードバック：[Bの出力]。対処して` | |

## 8. よくある失敗パターンと対策

| 失敗パターン | 対策 |
|------------|------|
| **キッチンシンクセッション**: 無関係なタスクが混在 | 無関係なタスク間で `/clear` |
| **繰り返し修正**: 同じ問題を何度も修正 | 2回修正しても改善しないなら `/clear` して新鮮な視点で再開 |
| **肥大化したCLAUDE.md**: 重要なルールが埋もれる | 容赦なく刈り込む。フックに変換 |
| **確認なしの信頼**: 正しく見えるが機能しない実装 | 常に検証（テスト・スクリプト・スクリーンショット）を提供 |
| **無限探索**: スコープなしで「調査して」 | 範囲を狭く指定するか、サブエージェントを使う |

## ハンズオン演習

### 演習 1: 4フェーズワークフローの実践

**目的**: 探索→計画→実装→コミットの4フェーズを実際に体験する

**前提条件**: テスト用のプロジェクトがあること

**手順**:

1. Claude Codeを起動し、`Shift+Tab` でPlan Modeに切り替える
2. **探索フェーズ**: プロジェクトの構造を分析させる
   ```text
   このプロジェクトの構造を分析して。主要なファイルとその役割を教えて。
   ```
3. **計画フェーズ**: 小さな改善（例: README.mdの改善）の計画を立てさせる
   ```text
   このプロジェクトのREADME.mdを改善する計画を作って。
   ```
4. `Shift+Tab` で通常モードに戻し、**実装フェーズ**を実行する
5. **コミットフェーズ**: 変更をコミットさせる

**期待される結果**: 4フェーズの流れを体験し、Plan Modeの切り替えに慣れる

### 演習 2: CLAUDE.mdの作成と最適化

**目的**: プロジェクトに最適なCLAUDE.mdを設計する

**前提条件**: テスト用のプロジェクトがあること

**手順**:

1. `/init` コマンドでCLAUDE.mdのスターターを生成する
2. 生成されたCLAUDE.mdを確認する
3. 以下の観点で改善する：
   - 不要な情報（コードを読めばわかること）を削除
   - テスト実行コマンドを追加
   - コードスタイルルールを追加
   - ブランチ命名規約を追加
4. `@import` 構文で外部ドキュメントを参照させる

**期待される結果**: 簡潔かつ効果的なCLAUDE.mdが完成する

### 演習 3: コンテキスト管理の実践

**目的**: コンテキスト管理コマンドの使い方に慣れる

**前提条件**: Claude Codeがインストール済みであること

**手順**:

1. Claude Codeを起動し、複数のファイルの内容を質問して読ませる
2. ステータスバーでコンテキスト使用率を確認する
3. `/compact` でコンテキストを圧縮する
4. 圧縮後のコンテキスト使用率の変化を確認する
5. `/clear` でコンテキストをリセットする
6. `/rename` でセッションに名前をつける
7. `claude --resume` で過去のセッションを確認する

**期待される結果**: コンテキスト管理コマンドの効果を体感し、適切なタイミングで使い分けられるようになる

## よくある質問

**Q: `/compact` はいつ実行すべきですか？**
A: コンテキスト使用率が70%に達する前に実行することを推奨します。使用率が80%を超えると品質劣化が顕著になります。また、タスクの切り替え時に `/compact 次のタスクに集中して` のようにフォーカスを絞って圧縮するのも効果的です。

**Q: CLAUDE.mdの適切な長さはどのくらいですか？**
A: 公式推奨は約500行以下です。長すぎるとコンテキストを消費し、重要な指示が埋もれます。各行が本当に必要かを定期的に見直し、Claudeが既に正しく行っていることは削除してください。

**Q: サブエージェントはいつ使うべきですか？**
A: コードベースの調査、ファイル検索、レビューなど、結果は必要だがプロセスの詳細は不要な場合に使います。メインのコンテキストウィンドウを汚染せずに作業を委譲できます。

**Q: ファンアウトパターンで並列度はどのくらいが適切ですか？**
A: APIのレート制限とローカルマシンのリソースに依存します。まず2-3件で検証し、問題がなければ並列度を上げてください。`--allowedTools` でツールを制限することで安全性を確保できます。

**Q: Claude Codeの出力品質が突然低下した場合、何を確認すべきですか？**
A: まずコンテキスト使用率を確認してください。70%を超えている場合は `/compact` または `/clear` を実行します。CLAUDE.mdが肥大化していないかも確認してください。

## まとめ

この章で学んだ重要ポイント：

- コンテキストウィンドウが最重要リソース。`/clear`・`/compact`・サブエージェントで積極管理する
- 検証可能なタスク設計（テスト・スクリーンショット・期待出力）がClaude Codeの品質を最大化する
- Plan Modeで探索→計画→実装→コミットの4フェーズを守り、「間違った問題の解決」を防ぐ
- CLAUDE.mdは短く・具体的に。不要な行は削除し、確実性が必要なものはフックに移行する
- `claude -p` とファンアウトパターンで大規模マイグレーションや分析をスケールできる
