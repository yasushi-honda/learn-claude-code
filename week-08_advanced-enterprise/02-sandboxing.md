# サンドボックス環境

> 対応する公式ドキュメント: [Sandboxing](https://code.claude.com/docs/en/sandboxing)

## 学習目標

- サンドボックスの概要と従来のパーミッションベースセキュリティとの違いを理解する
- ファイルシステム分離とネットワーク分離の仕組みを把握する
- macOS（Seatbelt）・Linux/WSL2（bubblewrap）でのサンドボックス設定ができる
- `settings.json`でのカスタムサンドボックス設定を実践できる
- サンドボックスとパーミッションの相互関係を理解する

## 概要

Claude Codeのネイティブサンドボックス機能は、エージェント実行のセキュリティを高めながら、定数的なパーミッションプロンプトの必要性を減らします。Bashコマンドごとに承認を求める代わりに、サンドボックスは最初に明確な境界を設定し、その境界内でClaude Codeがより自由に作業できるようにします。

サンドボックス化されたBashツールはOSレベルのプリミティブを使用してファイルシステムとネットワークの両方の分離を実施します。

> **重要**: 効果的なサンドボックスにはファイルシステム分離とネットワーク分離の**両方**が必要です。ネットワーク分離なしでは、攻撃者がSSHキーなどのファイルを流出させる可能性があります。ファイルシステム分離なしでは、ネットワークアクセスを得るためにシステムリソースに侵入する可能性があります。

## 本文

### サンドボックスが重要な理由

従来のパーミッションベースセキュリティの課題：

| 課題 | 説明 |
|------|------|
| **承認疲労** | 繰り返し「承認」をクリックすることで、実際に何を承認しているか注意が散漫になる |
| **生産性の低下** | 常時中断がワークフローを遅らせる |
| **自律性の制限** | 承認待ちのためClaudeが効率よく作業できない |

サンドボックスによる解決：

1. **明確な境界の定義**: アクセス可能なディレクトリとネットワークホストを明示的に指定
2. **パーミッションプロンプトの削減**: サンドボックス内の安全なコマンドは承認不要
3. **セキュリティの維持**: サンドボックス外へのアクセス試行は即座にブロック・通知
4. **自律性の実現**: 定義した制限内でClaudeがより独立して動作可能

### 仕組み

#### ファイルシステム分離

サンドボックス化されたBashツールはファイルシステムアクセスを特定のディレクトリに制限します：

| アクセス種別 | デフォルト動作 |
|------------|--------------|
| **デフォルト書き込み** | 現在の作業ディレクトリとそのサブディレクトリへの読み書きアクセス |
| **デフォルト読み取り** | コンピューター全体への読み取りアクセス（一部拒否ディレクトリを除く） |
| **ブロック済み** | 明示的な許可なしでの作業ディレクトリ外へのファイル変更 |
| **カスタマイズ可能** | 設定で許可・拒否パスを定義 |

> **重要**: これらの制限はOSレベルで実施されるため、Claude固有のファイルツールだけでなく、`kubectl`・`terraform`・`npm`などのすべてのサブプロセスコマンドにも適用されます。

#### ネットワーク分離

ネットワークアクセスはサンドボックス外で動作するプロキシサーバーで制御されます：

- **ドメイン制限**: 承認済みドメインのみアクセス可能
- **ユーザー確認**: 新しいドメインへのリクエストはパーミッションプロンプトをトリガー
- **カスタムプロキシ**: 送信トラフィックへのカスタムルールの実装が可能
- **包括的な適用**: コマンドが生成したすべてのスクリプト・プログラム・サブプロセスに制限が適用

#### OSレベルの実施

| OS | 使用技術 |
|---|---------|
| **macOS** | Seatbelt |
| **Linux** | [bubblewrap](https://github.com/containers/bubblewrap) |
| **WSL2** | bubblewrap（Linuxと同様） |
| **WSL1** | **非対応**（bubblewrapにはWSL2が必要なカーネル機能が必要） |

### 始め方

#### 前提条件

**macOS:** 組み込みのSeatbeltフレームワークを使用するため追加インストール不要。

**Linux・WSL2:** 必要パッケージをインストール：

```bash
# Ubuntu/Debian
sudo apt-get install bubblewrap socat

# Fedora
sudo dnf install bubblewrap socat
```

#### サンドボックスの有効化

`/sandbox`コマンドを実行：

```
/sandbox
```

メニューでサンドボックスモードを選択できます。必要な依存関係が不足している場合（LinuxのbubblewrapやsocatなどLinuxの場合）、プラットフォーム固有のインストール手順が表示されます。

### サンドボックスモード

#### Auto-allowモード

Bashコマンドはサンドボックス内で実行を試み、パーミッションなしで自動的に許可されます。サンドボックス化できないコマンド（許可されていないホストへのネットワークアクセスを必要とするものなど）は通常のパーミッションフローにフォールバックします。明示的に設定された許可/拒否ルールは常に尊重されます。

#### Regular permissionsモード

サンドボックス化されている場合でもすべてのBashコマンドが標準パーミッションフローを通過します。より多くの制御が可能ですが、より多くの承認が必要になります。

> **注意**: Auto-allowモードはパーミッションモード設定とは独立して動作します。「accept edits」モードでない場合でも、サンドボックス化されたBashコマンドは自動的に実行されます。

### サンドボックスの設定

`settings.json`でサンドボックスの動作をカスタマイズできます。

#### 特定のパスへの書き込みアクセスを許可

デフォルトではサンドボックス化されたコマンドは作業ディレクトリにのみ書き込めます。`kubectl`・`terraform`・`npm`などのサブプロセスが作業ディレクトリ外に書き込む必要がある場合：

```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["~/.kube", "//tmp/build"]
    }
  }
}
```

#### パスプレフィックスの規則

| プレフィックス | 意味 | 例 |
|-------------|------|---|
| `//` | ファイルシステムルートからの絶対パス | `//tmp/build` → `/tmp/build` |
| `~/` | ホームディレクトリからの相対パス | `~/.kube` → `$HOME/.kube` |
| `/` | 設定ファイルのディレクトリからの相対パス | `/build` → `$SETTINGS_DIR/build` |
| `./`または接頭辞なし | 相対パス | `./output` |

#### 読み取り・書き込みアクセスの拒否

```json
{
  "sandbox": {
    "filesystem": {
      "allowWrite": ["~/.kube"],
      "denyWrite": ["//etc", "~/.ssh"],
      "denyRead": ["~/.aws/credentials"]
    }
  }
}
```

#### 設定のマージ動作

複数の設定スコープで`allowWrite`・`denyWrite`・`denyRead`が定義されている場合、配列は**マージ**されます（置き換えではなく）。これにより、ユーザーとプロジェクトが上位スコープのパスを複製または上書きせずにリストを拡張できます。

### セキュリティへの貢献

#### プロンプトインジェクションへの保護

攻撃者がプロンプトインジェクションを通じてClaude Codeの動作を操作できても、サンドボックスはシステムの安全を維持します。

**ファイルシステム保護:**
- `~/.bashrc`などの重要な設定ファイルを変更できない
- `/bin/`のシステムレベルファイルを変更できない
- パーミッション設定で拒否されているファイルを読めない

**ネットワーク保護:**
- 攻撃者が制御するサーバーへのデータ流出ができない
- 許可されていないドメインから悪意のあるスクリプトをダウンロードできない
- 未承認のサービスへの予期しないAPI呼び出しができない
- 明示的に許可されていないドメインへの接続ができない

**監視と制御:**
- サンドボックス外へのすべてのアクセス試行がOSレベルでブロック
- 境界が試されると即座に通知
- 拒否・一度のみ許可・永続的に許可の選択が可能

#### 攻撃対象の削減

サンドボックスは以下からの潜在的な損害を制限します：
- 悪意のあるNPMパッケージなどの**悪意のある依存関係**
- セキュリティ脆弱性を持つ**侵害されたスクリプト**
- 危険なコマンドを実行させる**ソーシャルエンジニアリング**
- ClaudeをだましてClaudeが危険なコマンドを実行させる**プロンプトインジェクション**

### セキュリティ上の制限事項

#### ネットワークサンドボックスの制限

ネットワークフィルタリングは接続先ドメインを制限しますが、プロキシを通過するトラフィックは検査しません。ユーザーは許可するドメインが信頼できることを確認する責任があります。

> **注意**: `github.com`のような広いドメインを許可すると、データ流出のリスクがあります。また、[ドメインフロンティング](https://en.wikipedia.org/wiki/Domain_fronting)によりネットワークフィルタリングが回避される場合があります。

#### Unixソケット経由の権限昇格

`allowUnixSockets`設定は、サンドボックスバイパスにつながる可能性のある強力なシステムサービスへの意図しないアクセスを許可する可能性があります。例えば、`/var/run/docker.sock`へのアクセスを許可するとDockerソケットの悪用でホストシステムにアクセスできてしまいます。

#### ファイルシステム権限の昇格

過度に広いファイルシステム書き込み権限は権限昇格攻撃を可能にします。特に以下への書き込みを許可することは危険です：
- `$PATH`内の実行ファイルを含むディレクトリ
- システム設定ディレクトリ
- ユーザーのシェル設定ファイル（`.bashrc`・`.zshrc`）

#### Linux実装の強度

Linuxの実装は強力なファイルシステムとネットワーク分離を提供しますが、`enableWeakerNestedSandbox`モードはDocker環境内での動作を可能にするために大幅にセキュリティを弱めます。このオプションは追加の分離が別途実施される場合のみ使用してください。

### サンドボックスとパーミッションの関係

サンドボックスとパーミッションは補完的なセキュリティレイヤーです：

| 側面 | パーミッション | サンドボックス |
|------|--------------|-------------|
| **適用対象** | すべてのツール（Bash・Read・Edit・WebFetch・MCP等） | Bashコマンドとその子プロセスのみ |
| **実施レイヤー** | アプリケーションレベル（ツール実行前に評価） | OSレベル（ファイルシステムとネットワーク） |
| **設定場所** | `settings.json`の`permissions`セクション | `settings.json`の`sandbox`セクション |

#### 組み合わせて使用する

```json
{
  "permissions": {
    "allow": ["Bash(git *)"],
    "deny": ["Read(~/.ssh/*)", "Edit(~/.bashrc)"]
  },
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["~/.kube"],
      "denyWrite": ["~/.ssh", "~/.aws"]
    }
  }
}
```

### 高度な使用法

#### カスタムプロキシ設定

高度なネットワークセキュリティが必要な組織向け：

```json
{
  "sandbox": {
    "network": {
      "httpProxyPort": 8080,
      "socksProxyPort": 8081
    }
  }
}
```

カスタムプロキシを使用することで：
- HTTPSトラフィックの復号化と検査
- カスタムフィルタリングルールの適用
- すべてのネットワークリクエストのログ
- 既存のセキュリティインフラとの統合

が可能になります。

#### 既存のセキュリティツールとの統合

サンドボックス化されたBashツールは以下と連携して動作します：
- **パーミッションルール**: 多層防御のために組み合わせる
- **開発コンテナ**: devcontainersと組み合わせてさらなる分離を実現
- **企業ポリシー**: 管理設定でサンドボックス設定を強制

### オープンソース

サンドボックスランタイムはnpmパッケージとして公開されており、独自のエージェントプロジェクトで使用できます：

```bash
npx @anthropic-ai/sandbox-runtime <サンドボックス化するコマンド>
```

例えばMCPサーバーのサンドボックス化にも使用できます。詳細は[GitHubリポジトリ](https://github.com/anthropic-experimental/sandbox-runtime)を参照してください。

## まとめ

- サンドボックスはOSレベルでファイルシステムとネットワークの両方を分離し、プロンプトインジェクション攻撃からも保護する
- macOSではSeatbelt、Linux/WSL2ではbubblewrapを使用（WSL1は非対応）
- `/sandbox`コマンドでAuto-allowモードとRegular permissionsモードを選択できる
- `settings.json`の`sandbox.filesystem.allowWrite`で特定パスへの書き込みを許可できる
- サンドボックスとパーミッションは補完的なレイヤーであり、両方を組み合わせると効果的
- Unixソケットや過度に広い書き込み権限はセキュリティリスクとなるため注意が必要
- オープンソースのsandbox-runtimeパッケージで独自のエージェントにも適用可能
