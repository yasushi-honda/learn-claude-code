# サンドボックス環境

> **対応公式ドキュメント**: https://code.claude.com/docs/en/sandboxing
> **想定所要時間**: 約60分
> **難易度**: ★★★★☆

## この章の学習目標

この章を終えると、以下のことができるようになります：

1. ネイティブサンドボックスがOSレベルでファイルシステムとネットワークの両方を分離する仕組みを説明できる
2. macOS（Seatbelt）・Linux/WSL2（bubblewrap）でのサンドボックスを設定・有効化できる
3. `settings.json` でパス制御やネットワーク設定をカスタマイズできる
4. サンドボックスのセキュリティ上の制限事項を把握し、適切に対処できる

---

## 1. サンドボックスの概要と重要性

Claude Codeのネイティブサンドボックスは、**OSレベルのプリミティブ**を使用してBashコマンドのファイルシステムアクセスとネットワーク通信の両方を分離する機能です。従来のパーミッションベースセキュリティの課題を解決し、セキュリティを維持しながらClaude Codeの自律性を高めます。

### なぜ両方の分離が必要なのか

> **公式ドキュメントより**: 効果的なサンドボックスにはファイルシステム分離とネットワーク分離の**両方**が必要です。ネットワーク分離なしでは、攻撃者がSSHキーなどのファイルを流出させる可能性があります。ファイルシステム分離なしでは、ネットワークアクセスを得るためにシステムリソースに侵入する可能性があります。

### 従来のパーミッションベースセキュリティの課題

| 課題 | 説明 |
|------|------|
| **承認疲労** | 繰り返し「承認」をクリックすることで、実際に何を承認しているか注意が散漫になる |
| **生産性の低下** | 常時中断がワークフローを遅らせる |
| **自律性の制限** | 承認待ちのためClaudeが効率よく作業できない |

### サンドボックスによる解決

1. **明確な境界の定義**: アクセス可能なディレクトリとネットワークホストを明示的に指定
2. **パーミッションプロンプトの削減**: サンドボックス内の安全なコマンドは承認不要
3. **セキュリティの維持**: サンドボックス外へのアクセス試行は即座にブロック・通知
4. **自律性の実現**: 定義した制限内でClaudeがより独立して動作可能

## 2. 仕組みの詳細

### ファイルシステム分離

サンドボックス化されたBashツールはファイルシステムアクセスを特定のディレクトリに制限します：

| アクセス種別 | デフォルト動作 |
|------------|--------------|
| **デフォルト書き込み** | CWD（現在の作業ディレクトリ）以下への読み書きアクセス |
| **デフォルト読み取り** | CWD外を含むコンピューター全体への読み取りアクセス（一部拒否ディレクトリを除く） |
| **ブロック** | 明示的な許可なしでのCWD外への書き込み |
| **カスタマイズ可能** | 設定で許可・拒否パスを定義 |

> **重要**: これらの制限はOSレベルで実施されるため、Claude固有のツールだけでなく、`kubectl`・`terraform`・`npm` などのすべてのサブプロセスコマンドにも適用されます。

### ネットワーク分離

ネットワークアクセスはサンドボックス外で動作するプロキシサーバーで制御されます：

- **ドメイン制限**: 承認済みドメインのみアクセス可能
- **新ドメインのプロンプト**: 未承認のドメインへのリクエストはパーミッションプロンプトをトリガー
- **包括的な適用**: コマンドが生成したスクリプト・プログラム・サブプロセスにも制限が適用

### OSレベルの実施技術

| OS | 使用技術 | 備考 |
|---|---------|------|
| **macOS** | Seatbelt | 組み込みフレームワーク、追加インストール不要 |
| **Linux** | [bubblewrap](https://github.com/containers/bubblewrap) | `bubblewrap` と `socat` のインストールが必要 |
| **WSL2** | bubblewrap | Linuxと同様の手順 |
| **WSL1** | **非対応** | bubblewrapに必要なカーネル機能がWSL1にない |

## 3. セットアップと設定

### 前提条件のインストール

**macOS**: 追加インストール不要（Seatbeltは組み込み）

**Linux/WSL2**:

```bash
# Ubuntu/Debian
sudo apt-get install bubblewrap socat

# Fedora
sudo dnf install bubblewrap socat
```

### サンドボックスの有効化

Claude Code内で `/sandbox` コマンドを実行します：

```
/sandbox
```

メニューでサンドボックスモードを選択できます。必要な依存関係が不足している場合、プラットフォーム固有のインストール手順が表示されます。

### サンドボックスモード

#### Auto-allowモード

Bashコマンドはサンドボックス内で自動的に許可されます。サンドボックス化できないコマンド（許可されていないホストへのネットワークアクセス等）は通常のパーミッションフローにフォールバックします。明示的に設定された許可/拒否ルールは常に尊重されます。

#### Regular permissionsモード

サンドボックス化されている場合でも、すべてのBashコマンドが標準パーミッションフローを通過します。より多くの制御が可能ですが、承認の頻度は増えます。

> **注意**: Auto-allowモードはパーミッションモード設定とは独立して動作します。「accept edits」モードでない場合でも、サンドボックス化されたBashコマンドは自動的に実行されます。

### ファイルシステムのカスタマイズ

#### 特定パスへの書き込み許可

`kubectl`・`terraform`・`npm` などのツールがCWD外に書き込む必要がある場合：

```json
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["~/.kube", "//tmp/build"]
    }
  }
}
```

#### パスプレフィックスの規則

| プレフィックス | 意味 | 例 |
|-------------|------|---|
| `//` | ファイルシステムルートからの絶対パス | `//tmp/build` -> `/tmp/build` |
| `~/` | ホームディレクトリからの相対パス | `~/.kube` -> `$HOME/.kube` |
| `/` | 設定ファイルのディレクトリからの相対パス | `/build` -> `$SETTINGS_DIR/build` |
| `./` または接頭辞なし | 相対パス（CWDからの相対） | `./output` |

#### 読み取り・書き込みアクセスの拒否

```json
{
  "sandbox": {
    "filesystem": {
      "allowWrite": ["~/.kube"],
      "denyWrite": ["//etc", "~/.ssh"],
      "denyRead": ["~/.aws/credentials"]
    }
  }
}
```

#### 設定のマージ動作

複数の設定スコープ（ユーザー設定・プロジェクト設定等）で `allowWrite`・`denyWrite`・`denyRead` が定義されている場合、配列は**マージ**されます（置き換えではなく）。これにより、上位スコープのパスを複製・上書きせずにリストを拡張できます。

## 4. セキュリティへの貢献と制限事項

### プロンプトインジェクションからの保護

攻撃者がプロンプトインジェクションでClaude Codeの動作を操作できても、サンドボックスがシステムの安全を維持します：

**ファイルシステム保護:**
- `~/.bashrc` などの重要な設定ファイルの変更を防止
- `/bin/` のシステムレベルファイルの変更を防止
- 拒否設定されたファイルの読み取りを防止

**ネットワーク保護:**
- 攻撃者が制御するサーバーへのデータ流出を防止
- 未承認ドメインからの悪意のあるスクリプトのダウンロードを防止
- 明示的に許可されていないドメインへの接続を防止

**監視と制御:**
- サンドボックス外へのアクセス試行がOSレベルでブロック
- 境界が試されると即座に通知
- 拒否・一度のみ許可・永続的に許可の選択が可能

### セキュリティ上の制限事項

#### ネットワークフィルタリングの制限

ネットワークフィルタリングは接続先ドメインを制限しますが、**プロキシを通過するトラフィックの内容は検査しません**。許可するドメインが信頼できることを確認する責任はユーザーにあります。

> **注意**: `github.com` のような広いドメインを許可すると、データ流出のリスクがあります。また、[ドメインフロンティング](https://en.wikipedia.org/wiki/Domain_fronting)によりネットワークフィルタリングが回避される場合があります。

#### Unixソケット経由の権限昇格

`allowUnixSockets` 設定は慎重に使用してください。例えば、`/var/run/docker.sock` へのアクセスを許可すると、Dockerソケットの悪用でホストシステムにアクセスできてしまいます。

#### ファイルシステム権限の昇格

過度に広いファイルシステム書き込み権限は権限昇格攻撃を可能にします。特に以下への書き込みは危険です：

- `$PATH` 内の実行ファイルを含むディレクトリ
- システム設定ディレクトリ
- ユーザーのシェル設定ファイル（`.bashrc`・`.zshrc`）

#### enableWeakerNestedSandbox

Docker環境内で動作させるために `enableWeakerNestedSandbox` モードが用意されていますが、セキュリティを大幅に弱めます。このオプションは**追加の分離が別途実施される場合のみ**使用してください。

### エスケープハッチ

`dangerouslyDisableSandbox` はサンドボックスを無効化するエスケープハッチです。組織管理設定で `allowUnsandboxedCommands: false` を設定することで、このオプション自体を無効化できます。

## 5. 高度な使用法

### サンドボックスとパーミッションの組み合わせ

サンドボックスとパーミッションは補完的なセキュリティレイヤーです：

| 側面 | パーミッション | サンドボックス |
|------|--------------|-------------|
| **適用対象** | すべてのツール（Bash・Read・Edit・WebFetch・MCP等） | Bashコマンドとその子プロセスのみ |
| **実施レイヤー** | アプリケーションレベル（ツール実行前に評価） | OSレベル（ファイルシステムとネットワーク） |
| **設定場所** | `settings.json` の `permissions` セクション | `settings.json` の `sandbox` セクション |

```json
{
  "permissions": {
    "allow": ["Bash(git *)"],
    "deny": ["Read(~/.ssh/*)", "Edit(~/.bashrc)"]
  },
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": ["~/.kube"],
      "denyWrite": ["~/.ssh", "~/.aws"]
    }
  }
}
```

### カスタムプロキシ設定

高度なネットワークセキュリティが必要な組織向け：

```json
{
  "sandbox": {
    "network": {
      "httpProxyPort": 8080,
      "socksProxyPort": 8081
    }
  }
}
```

カスタムプロキシを使用することで、HTTPSトラフィックの復号化・検査、カスタムフィルタリングルールの適用、ネットワークリクエストのログ、既存のセキュリティインフラとの統合が可能になります。

### 互換性の注意

| ツール | 問題 | 対処法 |
|-------|------|--------|
| **watchman** | サンドボックスと非互換 | `--no-watchman` フラグを使用 |
| **Docker** | コンテナ内のサンドボックスが制限される | `excludedCommands` で除外するか、`enableWeakerNestedSandbox` を使用 |

### オープンソースのsandbox-runtime

サンドボックスランタイムはnpmパッケージとして公開されており、独自のエージェントプロジェクトで使用できます：

```bash
npx @anthropic-ai/sandbox-runtime <サンドボックス化するコマンド>
```

MCPサーバーのサンドボックス化にも使用できます。詳細は [GitHubリポジトリ](https://github.com/anthropic-experimental/sandbox-runtime) を参照してください。

## ハンズオン演習

### 演習 1: サンドボックスの有効化と動作確認

**目的**: サンドボックスを有効化し、ファイルシステム分離の動作を確認する

**前提条件**: macOSまたはLinux/WSL2環境、Claude Codeインストール済み

**手順**:

1. Claude Codeを起動する
2. `/sandbox` コマンドを実行し、Auto-allowモードを選択する
3. サンドボックスが有効化されたことを確認する
4. 以下のコマンドをClaudeに実行させる：
   ```
   CWD内にテストファイルを作成して: echo "test" > sandbox-test.txt
   ```
5. CWD外への書き込みを試みさせる：
   ```
   /tmp/outside-test.txt にファイルを書き込んで
   ```
6. CWD外への書き込みがブロックされることを確認する

**期待される結果**: CWD内の書き込みは成功し、CWD外の書き込みはサンドボックスによりブロックされる

### 演習 2: カスタムファイルシステム設定

**目的**: `settings.json` でサンドボックスの書き込み許可パスをカスタマイズする

**前提条件**: 演習1が完了していること

**手順**:

1. プロジェクトの `.claude/settings.json` を作成または編集する：
   ```json
   {
     "sandbox": {
       "enabled": true,
       "filesystem": {
         "allowWrite": ["//tmp/sandbox-test"],
         "denyRead": ["~/.ssh"]
       }
     }
   }
   ```
2. Claude Codeを再起動する
3. `/tmp/sandbox-test/` ディレクトリへの書き込みが許可されることを確認する
4. `~/.ssh/` ディレクトリの読み取りがブロックされることを確認する

**期待される結果**: カスタム設定に従ってアクセスが制御される

### 演習 3: サンドボックスとパーミッションの組み合わせ

**目的**: サンドボックスとパーミッションの両方を設定し、多層防御を実現する

**前提条件**: 演習1-2が完了していること

**手順**:

1. `.claude/settings.json` を以下のように設定する：
   ```json
   {
     "permissions": {
       "allow": ["Bash(git *)"],
       "deny": ["Read(./.env)", "Read(./.env.*)"]
     },
     "sandbox": {
       "enabled": true,
       "filesystem": {
         "denyWrite": ["~/.ssh", "~/.aws"]
       }
     }
   }
   ```
2. 各設定が正しく適用されることをテストする
3. パーミッションの拒否ルールとサンドボックスの拒否ルールの両方が機能することを確認する

**期待される結果**: パーミッションとサンドボックスの両方が補完的に動作し、多層防御が実現される

## よくある質問

**Q: サンドボックスを有効にすると、npmやpipのインストールは動作しますか？**
A: はい。`npm install` や `pip install` はCWD内の `node_modules` や `venv` に書き込むため、通常は問題ありません。ただし、グローバルインストール（`npm install -g`）はCWD外に書き込むためブロックされます。その場合は `allowWrite` でグローバルパスを追加するか、ローカルインストールを使用してください。

**Q: Auto-allowモードとRegular permissionsモードはどちらを使うべきですか？**
A: 日常の開発作業ではAuto-allowモードが生産性に優れます。セキュリティ監査や機密コードの作業時はRegular permissionsモードで個別承認を行うのが安全です。

**Q: WSL1でサンドボックスが使えないのはなぜですか？**
A: サンドボックスのLinux実装はbubblewrapに依存しており、bubblewrapは名前空間（namespaces）などのカーネル機能を必要とします。WSL1はWindowsカーネル上のエミュレーション層であり、これらの機能をサポートしていません。WSL2への移行を推奨します。

**Q: `dangerouslyDisableSandbox` を使う正当な理由はありますか？**
A: 特定のツール（例: Docker in Docker環境）で互換性の問題がある場合に限り、一時的な回避策として使用できます。組織管理者は `allowUnsandboxedCommands: false` でこのオプションを完全に無効化できます。

**Q: サンドボックスのオープンソースランタイムを自分のプロジェクトで使えますか？**
A: はい。`npx @anthropic-ai/sandbox-runtime <command>` で任意のコマンドをサンドボックス化できます。MCPサーバーの保護や、自作AIエージェントのセキュリティ強化に活用できます。

## まとめ

この章で学んだ重要ポイント：

- サンドボックスはOSレベル（macOS: Seatbelt、Linux/WSL2: bubblewrap）でファイルシステムとネットワークの両方を分離し、プロンプトインジェクション攻撃からも保護する
- `/sandbox` コマンドでAuto-allowモード（自動承認）とRegular permissionsモード（個別承認）を選択できる
- `settings.json` の `sandbox.filesystem` でパス制御（`allowWrite`・`denyWrite`・`denyRead`）をカスタマイズできる
- ネットワークフィルタリングはドメイン単位のみで、トラフィック内容は検査しない。ドメインフロンティングに注意が必要
- サンドボックスとパーミッションは補完的なレイヤーであり、両方を組み合わせることで多層防御が実現される
