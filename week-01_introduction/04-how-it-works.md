# エージェントループの仕組み

> 対応する公式ドキュメント: [How Claude Code Works](https://code.claude.com/docs/en/how-claude-code-works)

## 学習目標

- エージェントループの 3 フェーズ（gather context / take action / verify results）を説明できる
- Sonnet と Opus のモデルの使い分けを理解する
- 5 カテゴリのツール（ファイル操作・検索・実行・Web・コードインテリジェンス）を列挙できる
- チェックポイントとパーミッションモードの安全機構を理解する
- 効果的な使い方のヒントを実践できる

## 概要

Claude Code はエージェント型のアシスタントであり、ターミナルで動作します。コーディングに優れていますが、コマンドラインからできることすべて（ドキュメント作成、ビルド実行、ファイル検索、調査）に対応しています。このガイドでは、Claude Code の中核となるアーキテクチャ、内蔵された機能、そして効果的な使い方のヒントを解説します。

## 本文

### エージェントループ

Claude Code にタスクを与えると、3 つのフェーズを経て作業します：

```
gather context（コンテキスト収集）
      ↓
take action（アクション実行）
      ↓
verify results（結果検証）
      ↓
    繰り返し
```

これらのフェーズは互いに重なり合います。ファイルを検索してコードを理解し、編集して変更を加え、テストを実行して確認するという流れです。

**ループはタスクに応じて適応します：**

- コードベースへの質問 → コンテキスト収集のみ
- バグ修正 → 3 フェーズすべてを繰り返す
- リファクタリング → 広範な検証を伴う

あなたもこのループの一部です。いつでも割り込んで方向を変えたり、追加のコンテキストを提供したり、別のアプローチを試すよう指示できます。Claude Code は自律的に動作しながら、常にあなたの入力に対応します。

### モデル

Claude Code は Claude モデルを使用して、コードを理解しタスクについて推論します。Claude はあらゆる言語のコードを読み、コンポーネントの接続を理解し、目標を達成するために何を変更する必要があるかを把握します。

複数のモデルが利用可能で、それぞれにトレードオフがあります：

| モデル | 用途 |
|---|---|
| **Sonnet**（デフォルト） | ほとんどのコーディングタスクに適している |
| **Opus** | 複雑なアーキテクチャ決定など、より深い推論が必要な場合 |

セッション中に `/model` で切り替えるか、`claude --model <name>` で起動時に指定できます：

```bash
# セッション起動時にモデルを指定
claude --model claude-opus-4-6

# セッション内で切り替え
/model
```

### ツール

ツールは Claude Code をエージェント型にする要素です。ツールがなければ、Claude はテキストで返答するだけです。ツールがあることで、Claude はコードを読み、ファイルを編集し、コマンドを実行し、Web を検索し、外部サービスと連携することができます。

組み込みツールは大きく 5 つのカテゴリに分類されます：

| カテゴリ | Claude ができること |
|---|---|
| **File operations（ファイル操作）** | ファイルの読み込み、コードの編集、新規ファイルの作成、名前変更・整理 |
| **Search（検索）** | パターンでファイルを検索、正規表現でコンテンツを検索、コードベースの探索 |
| **Execution（実行）** | シェルコマンドの実行、サーバーの起動、テスト実行、git の使用 |
| **Web** | Web 検索、ドキュメントの取得、エラーメッセージの調査 |
| **Code intelligence（コードインテリジェンス）** | 型エラー・警告の確認、定義へのジャンプ、参照の検索（コードインテリジェンスプラグインが必要） |

**ツールチェーンの例（テスト失敗を修正する場合）：**

1. テストスイートを実行して失敗内容を確認
2. エラー出力を読み込む
3. 関連するソースファイルを検索
4. それらのファイルを読んでコードを理解
5. 問題を修正するためにファイルを編集
6. テストを再実行して検証

各ツール使用が新しい情報を提供し、次のステップに知らせます。これがエージェントループの実際の動作です。

### Claude Code がアクセスできるもの

`claude` をディレクトリで実行すると、Claude Code は以下にアクセスします：

| リソース | 説明 |
|---|---|
| **プロジェクト** | ディレクトリとサブディレクトリ内のファイル（許可があれば他の場所のファイルも） |
| **ターミナル** | 実行できるあらゆるコマンド（ビルドツール、git、パッケージマネージャー、スクリプト） |
| **Git 状態** | 現在のブランチ、未コミットの変更、最近のコミット履歴 |
| **CLAUDE.md** | プロジェクト固有の指示・規約・コンテキストが保存されたファイル |
| **Auto memory** | セッションをまたいで保存される学習内容（最初の 200 行が自動読み込み） |
| **設定した拡張機能** | MCP サーバー、スキル、サブエージェント、Chrome 連携など |

### セッション管理

Claude Code はセッション中の会話をローカルに保存します：

**セッションは独立しています：** 新しいセッションはフレッシュなコンテキストウィンドウで始まり、前のセッションの会話履歴はありません。Claude は Auto memory を使ってセッションをまたいで学習を永続化し、CLAUDE.md に独自の永続的な指示を追加できます。

**セッションの継続・フォーク：**

```bash
# 最近の会話を継続
claude --continue

# セッションを選択して再開
claude --resume

# 会話を分岐（別アプローチを試す）
claude --continue --fork-session
```

**ブランチをまたいだ作業：** セッションはディレクトリに紐づいています。ブランチを切り替えると Claude は新しいブランチのファイルを参照しますが、会話履歴はそのままです。

### コンテキストウィンドウ

Claude のコンテキストウィンドウには、会話履歴・ファイル内容・コマンド出力・CLAUDE.md・読み込んだスキル・システム命令が含まれます。作業が進むにつれてコンテキストが満たされますが、Claude は自動的にコンパクト化します。

コンテキストを確認するには `/context` を実行します。永続的なルールは会話履歴に頼るのではなく CLAUDE.md に書くことを推奨します。

```
/context    # コンテキスト使用状況を確認
/compact    # 会話を手動でコンパクト化
```

### 安全機構

Claude Code には 2 つの安全機構があります：

#### チェックポイント（ファイル変更の取り消し）

**すべてのファイル編集は取り消せます。** Claude がファイルを編集する前に、現在のファイルのスナップショットを保存します。問題が発生した場合は `Esc` を 2 回押して前の状態に巻き戻すか、Claude に「undo してください」と伝えます。

チェックポイントはセッションにローカルで、git とは別です。ファイル変更のみを対象とし、外部システム（データベース、API、デプロイ）への操作は取り消せません。

#### パーミッションモード

`Shift+Tab` を押してモードを切り替えます：

| モード | 動作 |
|---|---|
| **Default（デフォルト）** | ファイル編集とシェルコマンドの前に確認を求める |
| **Auto-accept edits（編集自動承認）** | ファイル編集は確認なし、コマンドは確認あり |
| **Plan mode（プランモード）** | 読み取り専用ツールのみ使用し、実行前に承認できるプランを作成 |

コマンドラインからプランモードで起動：

```bash
claude --permission-mode plan
```

特定のコマンドは `.claude/settings.json` で常に許可することができます（`npm test`、`git status` 等）。

### 効果的な使い方のヒント

#### Claude Code に使い方を聞く

```
> how do I set up hooks?
> what's the best way to structure my CLAUDE.md?
```

Claude Code は自身の使い方について説明できます。

#### 会話として使う

Claude Code は会話型です。完璧なプロンプトは不要です：

```
> Fix the login bug

[Claude が調査して修正を試みる]

> That's not quite right. The issue is in the session handling.

[Claude がアプローチを調整する]
```

最初の試みが正しくなくても、最初からやり直す必要はありません。反復できます。

#### 具体的に指示する

最初のプロンプトが具体的であればあるほど、修正が少なくなります：

```
> The checkout flow is broken for users with expired cards.
> Check src/payments/ for the issue, especially token refresh.
> Write a failing test first, then fix it.
```

「fix the login bug」のような曖昧なプロンプトも機能しますが、上記のような具体的なプロンプトは最初から成功することが多いです。

#### 検証できるものを与える

Claude はテストケースや期待する出力など、自分で作業を確認できる手段があると良いパフォーマンスを発揮します：

```
> Implement validateEmail. Test cases: 'user@example.com' → true,
> 'invalid' → false, 'user@.com' → false. Run the tests after.
```

#### 探索してから実装する

複雑な問題では、調査とコーディングを分けます。プランモードを使ってコードベースを先に分析します：

```
> Read src/auth/ and understand how we handle sessions.
> Then create a plan for adding OAuth support.
```

プランをレビューして会話で磨き上げてから、Claude に実装させます。

## まとめ

- エージェントループは **gather context → take action → verify results** の 3 フェーズで動作し、タスクが完了するまで繰り返す
- **Sonnet** が汎用モデル、**Opus** が複雑な推論向け。`/model` コマンドで切り替え可能
- 組み込みツールは 5 カテゴリ：ファイル操作・検索・実行・Web・コードインテリジェンス
- **チェックポイント**（Esc×2 で巻き戻し）と**パーミッションモード**（Shift+Tab で切り替え）が安全機構
- 具体的な指示・検証手段の提供・探索→実装の 2 段階が効果的な使い方のポイント
