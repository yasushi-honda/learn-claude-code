> **対応公式ドキュメント**: https://code.claude.com/docs/en/how-claude-code-works
> **想定所要時間**: 約60分
> **難易度**: ★★☆☆☆

## この章の学習目標

この章を終えると、以下のことができるようになります：

1. エージェントループの3フェーズ（gather context / take action / verify results）を説明できる
2. Claude Code が「エージェントハーネス」として何を提供しているか理解する
3. 5カテゴリのツール（ファイル操作・検索・実行・Web・コードインテリジェンス）を列挙し、それぞれの役割を説明できる
4. Sonnet と Opus のモデルの使い分けを判断できる
5. チェックポイント・パーミッションモード・コンテキスト管理の安全機構を理解し活用できる

## 1. エージェントハーネスとしての Claude Code

### エージェントハーネスとは

Claude Code は単なるチャットインターフェースではありません。**エージェントハーネス**として、AI モデル（Claude）にツール、コンテキスト管理、実行環境を提供するフレームワークです。

```
┌──────────────────────────────────────────────────┐
│              Claude Code (エージェントハーネス)     │
│                                                  │
│  ┌──────────┐  ┌───────────────┐  ┌───────────┐ │
│  │ ツール群   │  │ コンテキスト   │  │  実行環境   │ │
│  │ (5カテゴリ) │  │  管理         │  │ (サンド    │ │
│  │           │  │ (ウィンドウ)   │  │  ボックス)  │ │
│  └──────────┘  └───────────────┘  └───────────┘ │
│                      ↕                           │
│  ┌──────────────────────────────────────────┐    │
│  │         Claude AI モデル                  │    │
│  │    (Sonnet: デフォルト / Opus: 高度な推論)  │    │
│  └──────────────────────────────────────────┘    │
└──────────────────────────────────────────────────┘
```

> **公式ドキュメントより**: Claude Code はコーディングに優れていますが、コマンドラインからできることすべて -- ドキュメント作成、ビルド実行、ファイル検索、調査 -- に対応しています。

### エージェントループの3フェーズ

Claude Code にタスクを与えると、以下の3フェーズを繰り返しながら作業を進めます。

#### Phase 1: Gather Context（コンテキスト収集）

タスクに必要な情報を集めるフェーズです。

- プロジェクトのファイル構造を確認
- 関連するソースファイルを読み込む
- Git の状態（ブランチ、未コミットの変更）を確認
- CLAUDE.md からプロジェクト固有の指示を取得
- 必要に応じて Web 検索でドキュメントを参照

#### Phase 2: Take Action（アクション実行）

集めたコンテキストに基づいてアクションを実行するフェーズです。

- ファイルの編集（新規作成・変更・削除）
- シェルコマンドの実行（ビルド・テスト・lint）
- Git 操作（コミット・ブランチ作成・プッシュ）

#### Phase 3: Verify Results（結果検証）

アクションの結果を検証するフェーズです。

- テストを実行して変更が正しいか確認
- lint / 型チェックでコード品質を検証
- コマンドの出力を分析してエラーがないか確認
- 必要に応じて追加の修正を行う

```
  タスク受領
      │
      ▼
  ┌─────────────────────┐
  │ Gather Context      │ ← ファイル読込・検索・Git状態確認
  │ (コンテキスト収集)    │
  └─────────┬───────────┘
            │
            ▼
  ┌─────────────────────┐
  │ Take Action         │ ← ファイル編集・コマンド実行
  │ (アクション実行)      │
  └─────────┬───────────┘
            │
            ▼
  ┌─────────────────────┐
  │ Verify Results      │ ← テスト実行・結果分析
  │ (結果検証)           │
  └─────────┬───────────┘
            │
            ▼
       完了？ ─── No ──→ Gather Context へ戻る
        │
       Yes
        │
        ▼
    タスク完了
```

### ループの適応性

ループはタスクの種類に応じて自動的に適応します。

| タスク | 実行されるフェーズ |
|---|---|
| コードベースへの質問 | コンテキスト収集のみ |
| 単純なバグ修正 | 3フェーズを1-2回 |
| 大規模リファクタリング | 3フェーズを複数回（広範な検証を伴う） |
| テスト駆動開発 | テスト作成 → 実装 → 検証を繰り返す |

### あなたもループの一部

重要なのは、**あなたもこのループの一部である**ということです。Claude Code が作業している最中でも、いつでも割り込んで方向を変えたり、追加のコンテキストを提供したり、別のアプローチを試すよう指示できます。

```
> Fix the login bug

[Claude が調査して修正を試みる]

> That's not quite right. The issue is in the session handling.

[Claude がアプローチを調整する]
```

最初の試みが正しくなくても、最初からやり直す必要はありません。会話として反復できます。

## 2. モデル

Claude Code は Claude モデルを使用して、コードを理解しタスクについて推論します。

### 利用可能なモデル

| モデル | 用途 | 特徴 |
|---|---|---|
| **Sonnet**（デフォルト） | ほとんどのコーディングタスク | 高速・コスト効率が高い・十分な精度 |
| **Opus** | 複雑なアーキテクチャ決定・深い推論 | より高精度だがコストが高い |

### モデルの切り替え

```bash
# 起動時にモデルを指定
claude --model claude-opus-4-6

# セッション内で切り替え
/model
```

> **実践的なアドバイス**: 日常的なコーディング作業では Sonnet で十分です。複雑なアーキテクチャ設計や、難解なバグのデバッグなど、深い推論が必要な場面でのみ Opus に切り替えることを推奨します。

## 3. ツール

ツールは Claude Code を「エージェント型」にする核心です。ツールがなければ、Claude はテキストで返答するだけです。ツールがあることで、Claude はコードを読み、ファイルを編集し、コマンドを実行し、Web を検索し、外部サービスと連携することができます。

### 5つのツールカテゴリ

| カテゴリ | Claude ができること | 具体例 |
|---|---|---|
| **File operations** | ファイルの読み書き・作成・整理 | ソースコード編集、設定ファイル更新 |
| **Search** | パターンでファイル検索、正規表現で内容検索 | 関数定義の検索、依存関係の調査 |
| **Execution** | シェルコマンド実行 | テスト実行、ビルド、git、パッケージ管理 |
| **Web** | Web 検索、ドキュメント取得 | エラーメッセージの調査、API ドキュメント参照 |
| **Code intelligence** | 型エラー確認、定義ジャンプ、参照検索 | TypeScript の型エラー検出、シンボル参照 |

> **注意**: Code intelligence カテゴリはコードインテリジェンスプラグインが必要です。

### ツールチェーンの実例

テスト失敗を修正する場合のツールチェーン例です。

```
1. [Execution]  テストスイートを実行して失敗を確認
2. [Search]     エラー出力に関連するソースファイルを検索
3. [File ops]   該当ファイルを読み込んでコードを理解
4. [File ops]   問題を修正するためにファイルを編集
5. [Execution]  テストを再実行して検証
6. [File ops]   修正が不十分なら追加の編集
7. [Execution]  最終テスト実行
```

各ツール使用が新しい情報を提供し、次のステップに知らせます。これがエージェントループの実際の動作です。

## 4. Claude Code がアクセスできるもの

`claude` をディレクトリで実行すると、Claude Code は以下のリソースにアクセスします。

| リソース | 説明 |
|---|---|
| **プロジェクトファイル** | ディレクトリとサブディレクトリ内のファイル（許可があれば他の場所のファイルも） |
| **ターミナル** | 実行できるあらゆるコマンド（ビルドツール、git、パッケージマネージャー、スクリプト） |
| **Git 状態** | 現在のブランチ、未コミットの変更、最近のコミット履歴 |
| **CLAUDE.md** | プロジェクト固有の指示・規約・コンテキストが保存されたファイル |
| **Auto memory** | セッションをまたいで保存される学習内容（最初の200行が自動読み込み） |
| **MCP サーバー** | 設定された外部サービス連携（DB、API、チャットツール等） |

## 5. セッション管理

### セッションの独立性

新しいセッションはフレッシュなコンテキストウィンドウで始まり、前のセッションの会話履歴は含まれません。セッション間の知識の永続化は、Auto memory と CLAUDE.md によって行われます。

### セッションの継続・分岐

```bash
# 直前の会話を継続
claude --continue    # または claude -c

# 過去の会話を選択して再開
claude --resume      # または claude -r

# 会話を分岐（別アプローチを試す場合）
claude --continue --fork-session
```

> **Tip**: `--fork-session` は、現在の会話のコピーを作成して別のアプローチを試したいときに便利です。元の会話には影響しません。

### コンテキストウィンドウ

Claude のコンテキストウィンドウには、会話履歴・ファイル内容・コマンド出力・CLAUDE.md・読み込んだスキル・システム命令が含まれます。作業が進むにつれてコンテキストが満たされます。

#### コンテキスト管理のコマンド

```
/context    # コンテキスト使用状況を確認
/compact    # 会話を手動でコンパクト化（要約して圧縮）
```

> **公式ドキュメントより**: コンテキストが上限に近づくと、Claude Code は自動的にコンパクト化を行います。永続的なルールは会話履歴に頼るのではなく CLAUDE.md に記載することを推奨します。

## 6. 安全機構

Claude Code には、予期しない変更からプロジェクトを保護するための安全機構があります。

### チェックポイント（ファイル変更の取り消し）

**すべてのファイル編集は取り消せます。** Claude がファイルを編集する前に、現在のファイルのスナップショットを保存します。

- `Esc` を2回押す: 最後のファイル変更を巻き戻し
- Claude に「undo してください」と伝える

> **重要**: チェックポイントはセッションにローカルで、git とは別です。ファイル変更のみを対象とし、外部システム（データベース、API コール、デプロイ）への操作は取り消せません。

### パーミッションモード

`Shift+Tab` を押してモードを切り替えます。

| モード | ファイル編集 | コマンド実行 | 用途 |
|---|---|---|---|
| **Default** | 確認あり | 確認あり | 初心者・慎重な作業 |
| **Auto-accept edits** | 確認なし | 確認あり | 日常的な開発 |
| **Plan mode** | 読み取り専用 | 読み取り専用 | 調査・設計フェーズ |

コマンドラインからプランモードで起動:

```bash
claude --permission-mode plan
```

#### 許可リスト

特定のコマンドを `.claude/settings.json` で常に許可することができます。

```json
{
  "permissions": {
    "allow": [
      "npm test",
      "npm run lint",
      "git status",
      "git diff"
    ]
  }
}
```

許可リストに追加したコマンドは、Default モードでも確認なしで実行されます。

## ハンズオン演習

### 演習 1: エージェントループを観察する

**目的**: Claude Code のエージェントループ（3フェーズ）が実際に動作する様子を観察する
**前提条件**: Claude Code がインストール済み、テスト用プロジェクトがある

**手順**:

1. テスト用プロジェクトを準備する
   ```bash
   mkdir -p ~/agent-loop-test
   cd ~/agent-loop-test
   git init
   echo 'function add(a, b) { return a + b; }' > math.js
   echo 'module.exports = { add };' >> math.js
   git add . && git commit -m "initial commit"
   ```

2. Claude Code を起動し、テスト作成を依頼する
   ```bash
   claude
   ```
   ```
   > write tests for math.js, run them, and fix any failures
   ```

3. Claude Code の動作を観察し、以下を記録する:
   - **Gather Context**: どのファイルを読み込んだか
   - **Take Action**: どのファイルを作成/編集したか
   - **Verify Results**: どのコマンドを実行して検証したか

4. ループが何回繰り返されたか数える

**期待される結果**: Claude Code が math.js を読み、テストファイルを作成し、テストを実行し、必要に応じて修正するという流れを観察できる

### 演習 2: パーミッションモードを切り替える

**目的**: 3つのパーミッションモードの違いを体験する
**前提条件**: 演習 1 のプロジェクトが存在する

**手順**:

1. Claude Code を起動する

2. Default モードで変更を依頼する
   ```
   > add a multiply function to math.js
   ```
   確認プロンプトが表示されることを確認する

3. `Shift+Tab` を押して Auto-accept edits モードに切り替える

4. もう一つの変更を依頼する
   ```
   > add a subtract function to math.js
   ```
   ファイル編集が確認なしで実行されることを確認する

5. `Shift+Tab` を押して Plan mode に切り替える

6. 変更を依頼する
   ```
   > add a divide function to math.js
   ```
   プランのみが提示され、実際の変更は行われないことを確認する

**期待される結果**: 各モードで異なる動作（確認あり/なし/プランのみ）を体験できる

### 演習 3: コンテキスト管理を体験する

**目的**: コンテキストウィンドウの管理方法を学ぶ
**前提条件**: Claude Code がインストール済み

**手順**:

1. Claude Code を起動する

2. いくつかの質問をする
   ```
   > what does this project do?
   > explain the folder structure
   > what tests exist?
   ```

3. コンテキスト使用状況を確認する
   ```
   /context
   ```

4. コンパクト化を実行する
   ```
   /compact
   ```

5. 再度コンテキスト使用状況を確認する
   ```
   /context
   ```
   コンパクト化前後でコンテキスト使用量が減少していることを確認する

6. 会話をクリアする
   ```
   /clear
   ```

**期待される結果**: `/context` でコンテキスト使用量を確認でき、`/compact` で圧縮されることを体験できる

## よくある質問

**Q: Sonnet と Opus、どちらを使うべきですか？**
A: 日常的なコーディング作業（実装、テスト、リファクタリング）では **Sonnet** で十分です。複雑なアーキテクチャ設計、難解なバグのデバッグ、複数システム間の整合性を要する判断など、深い推論が必要な場面でのみ **Opus** に切り替えることを推奨します。

**Q: コンテキストウィンドウが満杯になるとどうなりますか？**
A: Claude Code は自動的にコンパクト化（会話の要約・圧縮）を行います。ただし、品質の劣化を防ぐため、コンテキスト使用率が70%に達する前に手動で `/compact` を実行するか、`/clear` で新しい会話を始めることを推奨します。

**Q: チェックポイントは Git のコミットと同じですか？**
A: いいえ。チェックポイントは Claude Code のセッション内のファイルスナップショットで、Git とは独立しています。セッションが終了するとチェックポイントは失われます。永続的な変更の管理には Git を使ってください。

**Q: Plan mode はどのような場面で使いますか？**
A: 大規模な変更を行う前の調査・設計フェーズで使います。Claude Code にコードベースを分析させ、変更計画を立てさせた後、プランをレビューしてから実行に移すことで、予期しない変更を防げます。

**Q: 外部システムへの操作（DB更新、API呼び出し等）は取り消せますか？**
A: いいえ。チェックポイントはローカルファイルの変更のみを対象としています。データベース操作、API 呼び出し、デプロイなどの外部システムへの操作は取り消せません。重要な外部操作を行う前は、Plan mode で計画を確認することを推奨します。

## まとめ

この章で学んだ重要ポイント：

- Claude Code は **エージェントハーネス** として、AI モデルにツール・コンテキスト管理・実行環境を提供する
- エージェントループは **gather context → take action → verify results** の3フェーズで動作し、タスク完了まで繰り返す
- **Sonnet** が汎用モデル（デフォルト）、**Opus** が複雑な推論向け。`/model` で切り替え可能
- 組み込みツールは5カテゴリ: **File operations / Search / Execution / Web / Code intelligence**
- **チェックポイント**（`Esc` x 2 で巻き戻し）と **パーミッションモード**（`Shift+Tab` で切り替え）が安全機構
- コンテキスト管理: `/context` で確認、`/compact` で圧縮、`/clear` でリセット
